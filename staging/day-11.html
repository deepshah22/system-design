Database Indexing Strategies
How to Find One Row in a Billion, Instantly

Yesterday Database Dave learned he can't have consistency and availability both, during a partition. Today his problem is smaller and far more common: he has one billion rows and a customer waiting. Read them all and the customer times out; find the right one in the blink of an eye and they never notice the billion. The difference between those two worlds is a single data structure — and knowing exactly when it saves you and when it quietly taxes every write is what separates people who "add an index" from people who design for scale.

You look up a word in a 2,000-page dictionary. You do not start on page 1 and read every word until you hit the one you want — you'd still be reading tomorrow. You flip roughly to where it should be, narrow in, and land on it in seconds. Now: why does your database, given a billion rows and asked for exactly one, so often do the equivalent of reading from page 1?

The answer is that a database only knows how to flip-to-roughly-the-right-place if someone built it the dictionary's thumb-tabs first. That structure is called an index, and building it is not automatic — it's a decision, with a cost. Today is about making that decision like an engineer instead of guessing.

Visual Memory Anchor

🗂️📖

Picture this

The back of a thick textbook. You want every page that mentions "mitochondria." You do not reread the whole book — you flip to the index at the back, run your finger down the alphabetized list to M, and it hands you the exact page numbers: 44, 91, 305. The index is a small, sorted, separate copy of just the keyword and a pointer to where the real content lives. A database index is that, mechanized: a sorted structure holding just the column you search by, plus a pointer to the full row. Recall the finger landing on "mitochondria, p.44" and you've recalled the entire concept.

The Story

Database Dave keeps his customer ledger in one giant bound book — one row per customer, in the order people signed up. It worked fine at 500 customers. Now there are 50 million, and every time someone asks "what's the balance for account #8,300,412?", poor Dave starts at page one and turns pages until he finds it. On a bad day that's 50 million page-turns for a single question. Customers are timing out at the counter.

A librarian friend watches this once, winces, and says: "Dave, build me a card catalog." Overnight they create a second, slim set of cards — just the account number and the page it's on — and, crucially, keep those cards sorted by account number. Now when someone asks for #8,300,412, Dave doesn't read the ledger at all. He opens the catalog to the middle, sees he's too high, jumps to the middle of the lower half, too low, splits again… and in about 26 hops he's holding the exact card that says "page 4,150,206." One flip to that page, done. Fifty million turns became twenty-six.

Dave is thrilled — until the librarian adds the catch. "Every time a new customer signs up, or an account number changes, you now have to slot a new card into the right sorted spot in the catalog too. The ledger got faster to read and a little slower to write. And if you build me twelve different catalogs — one sorted by name, one by city, one by signup date — every single write has to update all twelve." That trade — pay a little on every write to save enormously on reads — is the entire art of indexing.

What should stay in your head
An index is a separate, always-sorted copy of one or more columns plus a pointer to the full row. It turns "scan every row" (O(n)) into "flip straight to it" (O(log n)) for reads — and in exchange, every write must also update every index that touches the changed columns. You are trading write speed and disk space for read speed.

How Indexes Actually Work

The default is a full table scan — and it's O(n)

With no index, the only way a database can find rows matching WHERE id = 8300412 is to read every row and check. That's a full table scan: linear time, O(n). At 50 rows it's invisible. At 50 million rows, sitting on disk, it's the difference between a query that returns in 2 ms and one that takes several seconds and pins a CPU. Reads that scan are also selfish — they evict useful data from cache and starve every other query sharing the machine.

A B-tree turns O(n) into O(log n)

The workhorse index in nearly every relational database (Postgres, MySQL/InnoDB, SQL Server, Oracle) is a B-tree — more precisely a B+tree. Picture the card catalog as a shallow, bushy tree: a root card points to a handful of middle cards, each of which points to more, until the bottom "leaf" level holds the actual sorted keys and their row pointers. Because each node holds hundreds of keys, the tree is astonishingly shallow — a B+tree over a billion rows is typically only 3–4 levels deep. Finding any key means reading ~4 nodes, not a billion rows. That's the whole magic: each hop down the tree throws away most of the remaining data.

1.Start at the root node (in memory). Compare your key to its sorted keys; follow the one pointer whose range contains it.
2.Land on an internal node. Compare again, follow the right pointer. Repeat — each step drops you one level and discards ~99% of what's left.
3.Arrive at a leaf node holding the exact key and a pointer to the full row (or, for a clustered index, the row itself).
4.Follow the pointer, read the row, return it. Total work: a handful of node reads regardless of table size.

B+trees also keep their leaves linked in sorted order, which is why the same index that answers WHERE id = ? also makes WHERE id BETWEEN 100 AND 200, ORDER BY id, and WHERE id > 500 LIMIT 20 fast — you find the start, then walk the leaf chain. Range queries and sorting come free with a B-tree, which is exactly why it beats a hash index in practice.

The Kinds of Index (and When to Reach for Each)

B-tree / B+tree

The default. Sorted, so it serves equality and ranges, sorting, and prefix matches. Your first choice unless you have a specific reason not to.

Great for: =, <, >, BETWEEN, ORDER BY, LIKE 'abc%'

Hash index

Hashes the key to a bucket — O(1) equality lookups, often faster than a B-tree for exact matches. But it's unsorted, so it can't do ranges or ordering at all.

Great for: exact = only. Redis, in-memory tables.

Composite (multi-column)

One index over several columns, e.g. (user_id, created_at). Obeys the leftmost-prefix rule — column order is a design decision, not a detail.

Great for: queries filtering on the leading column(s).

Covering index

Includes every column the query needs, so the database answers entirely from the index and never touches the table. An "index-only scan" — the fastest a read gets.

Great for: hot read paths where you control the exact query.

Clustered index

The table is the index — rows are physically stored in index order (InnoDB's primary key). One per table. Range scans on it are blisteringly fast; secondary indexes point back through it.

Great for: the primary key / most common range column.

Specialized

Partial (index only rows matching a condition), full-text (words in documents), GIN/GiST (arrays, JSON, geospatial), LSM-trees (write-optimized, in Cassandra/RocksDB).

Great for: search, JSON, geo, and write-heavy stores.

The leftmost-prefix rule — the composite-index trap

A composite index on (user_id, created_at) is a catalog sorted first by user_id, and within each user, by created_at. That means it can answer:

Query filters on…Uses index (user_id, created_at)?
user_id = 7✅ Yes — leading column
user_id = 7 AND created_at > '2026-01-01'✅ Yes — full prefix, ideal
created_at > '2026-01-01' (no user_id)❌ No — skips the leading column, like searching a phone book by first name

Column order is everything. Put the column you always filter by first, and the one you range or sort by second. Getting this wrong is the single most common reason an index "exists but isn't being used."

The Hidden Cost Nobody Mentions

Indexes are not free lunch — they are a read-write trade, and juniors only see the read half:

Write amplification.Every INSERT, UPDATE, or DELETE must also update every index touching the changed columns — keeping each one sorted. Five indexes means one logical write becomes six physical writes. On write-heavy tables this dominates.
Disk & memory.An index is a whole extra copy of its columns. Indexes can easily exceed the size of the table itself, and they compete with your data for precious RAM/cache.
Low cardinality is a trap.Indexing a column with few distinct values (e.g. is_active, a boolean) barely narrows anything — the planner will often ignore it and scan anyway, because reading half the table through an index is slower than a straight scan. Index high-cardinality/selective columns.
The planner decides, not you.You create an index; the query optimizer chooses whether to use it based on estimated selectivity and its statistics. An index the planner never picks is pure cost. EXPLAIN is how you check.

Rule of thumb: index the columns you filter, join, and sort on frequently and selectively; don't reflexively index everything; and on an insert-heavy table, treat each extra index as a tax on ingestion you must justify.

Common Mistakes & Misconceptions

The misconceptionThe reality
"Indexes always make the database faster."They make reads faster and writes slower. On a write-heavy table, over-indexing is a net loss.
"Just add an index on every column I query."The planner may never use most of them; each is pure write + storage cost. Composite indexes often replace several single-column ones.
"An index on (a, b) also covers queries on b alone."No — leftmost-prefix rule. It covers a and (a, b), not b by itself.
"More index = better; index the boolean flags too."Low-cardinality columns barely filter anything; the planner usually ignores the index and scans.
"I created the index, so my query is fast now."Only if the optimizer chooses it. Verify with EXPLAIN / EXPLAIN ANALYZE — an unused index is dead weight.

Active Recall

❓ You add an index and reads get faster — name the two costs you just silently signed up for, and one situation where the index makes things worse overall. (click to reveal)
Cost 1: write amplification — every insert/update/delete must also keep this index sorted, so writes get slower. Cost 2: storage/memory — the index is an extra sorted copy of its columns competing for disk and cache. It makes things worse overall on a write-heavy table (the write tax outweighs the read win) or on a low-cardinality column (it barely filters, so the planner scans anyway and you paid the write cost for nothing).

Feynman Test

Explain to a 10-year-old how a database can find one specific customer out of a billion almost instantly, when there are way too many to check one by one. Write it below, then compare.

"It's like the little list at the back of a big book. If you want every page about dinosaurs, you don't read the whole book — you flip to the back where all the words are in ABC order, run your finger to 'dinosaur,' and it tells you the exact pages. The database keeps a tidy sorted list like that of just the thing you're searching for, plus a note saying where the full information is. Because the list is sorted, it can keep cutting it in half — 'too high, too low, too high' — and get to the answer in just a few jumps instead of checking a billion things. The only catch: every time a new customer is added, it has to squeeze them into the right spot on that sorted list too, which takes a tiny bit of extra work."

Real-World Engineering

🐘 PostgreSQL

Ships B-tree, hash, GiST, GIN, BRIN and more. GIN indexes power full-text search and jsonb queries; partial indexes (WHERE status='active') keep hot indexes tiny. Engineers live in EXPLAIN ANALYZE to confirm the planner actually chose their index.

🐬 MySQL / InnoDB

The primary key is a clustered index — rows are physically stored in PK order — so choosing a monotonic PK (not a random UUID) keeps inserts sequential and pages full. Every secondary index stores the PK as its row pointer, so a fat PK bloats every other index.

🪶 Cassandra / RocksDB

Use LSM-trees instead of B-trees: writes go to an in-memory sorted buffer and are flushed as immutable sorted files, later compacted. This flips the trade — writes become extremely cheap and sequential, at the cost of read amplification and background compaction. The right choice for write-heavy, high-ingest systems.

🔎 Elasticsearch

Built entirely around the inverted index — a map from each term to the list of documents containing it — which is the same "sorted keyword → pointers" idea you saw in the textbook analogy, specialized for full-text search at massive scale (we'll design one on Day 55).

Flashcards (Spaced Repetition)

Q: In one sentence, what is a database index and what does it trade?
A: A separate, always-sorted copy of one or more columns plus a pointer to the full row; it trades slower writes and extra storage for much faster reads (O(n) scan → O(log n) lookup).
Q: Why is a B+tree over a billion rows only ~4 levels deep, and why does that matter?
A: Each node holds hundreds of keys, so the branching factor is huge — depth grows as log-base-(hundreds) of n. A lookup reads ~4 nodes instead of a billion rows, and linked leaves make ranges/sorting fast for free.
Q: State the leftmost-prefix rule for a composite index on (a, b, c).
A: It can serve queries filtering on a, (a,b), or (a,b,c) — a leading prefix — but not b alone or c alone. Column order is a design decision: put the always-filtered column first.
Q: Give two situations where adding an index is a net negative.
A: (1) A write-heavy table — write amplification outweighs the read gain. (2) A low-cardinality column (e.g. a boolean) — it barely filters, so the planner ignores it and you paid the write cost for nothing.
Q: What's a covering index?
A: An index containing every column a query needs, so the database answers entirely from the index without reading the table (an "index-only scan") — the fastest a read gets.

Memory Test — Answer Without Looking Up

  • Explain why a full table scan is O(n) and how a B-tree makes the same lookup O(log n) — including why the tree is so shallow.
  • You have an index on (user_id, created_at). Which of these use it: filtering by user_id? by created_at alone? by both? Why?
  • Name two costs of an index and one type of column you should not index. Explain the cardinality reasoning.
  • When would you choose a hash index over a B-tree — and what do you give up?
  • Why do Cassandra and RocksDB use LSM-trees instead of B-trees, and what trade does that flip?
  • You added an index but the query is still slow. What single command do you run to find out why, and what are you looking for?
If you only remember one thing: an index is a sorted card catalog that turns "read a billion rows" into "hop down four nodes" — but you pay for every read shortcut with a slower write and more disk, so index the columns you filter/join/sort on selectively, put the leading column of a composite index first, and always confirm with EXPLAIN that the planner actually took the shortcut you built.

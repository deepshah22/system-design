Database Indexing Strategies
How to Find One Row in a Billion, Instantly

A database with a billion rows and a customer waiting has two ways to answer a question: read all billion rows one by one, or flip straight to the answer in a handful of steps. The difference between those two worlds is a single data structure called an index — and knowing exactly when it saves you and when it quietly taxes every write is what separates people who "just add an index" from people who design for scale.

Introduction

Imagine looking up a single word in a 2,000-page dictionary. You would never start on page 1 and read every word until you reached the one you wanted — you would be reading until tomorrow. Instead you flip roughly to where the word should be, narrow in, and land on it in seconds. That only works because the dictionary is sorted and has thumb-tabs.

A database faces the same problem, but with millions or billions of rows. By default, when you ask it "give me the customer with account #8,300,412," it has no thumb-tabs. It starts at the first row and checks every single one until it finds a match — the database equivalent of reading the dictionary cover to cover. At small scale you never notice. At large scale it is the difference between a page that loads instantly and one that times out.

An index is the thumb-tabs. It is a separate, always-sorted structure that lets the database jump almost directly to the rows you asked for, instead of scanning everything. Today we build a precise, engineer's understanding of how indexes work, the different kinds and when to use each, and — crucially — the hidden cost that makes "just add an index to everything" a mistake.

Concept Explanation

1. The default is a full table scan — and it is O(n)

With no index, the only way a database can satisfy WHERE id = 8300412 is to read every row and compare. That is a full table scan: linear time, O(n). At 50 rows it is invisible; at 50 million rows sitting on disk it is the difference between a query returning in 2 milliseconds and one taking several seconds while pinning a CPU core. Scans are also selfish — they pull huge amounts of data through memory, evicting other queries' useful cached pages and slowing down everything else sharing the machine.

2. A B-tree turns O(n) into O(log n)

The workhorse index in nearly every relational database (PostgreSQL, MySQL/InnoDB, SQL Server, Oracle) is a B-tree — more precisely a B+tree. Picture a shallow, bushy tree: a root node points to a handful of middle nodes, each of which points to more, until the bottom "leaf" level holds the actual sorted keys and their row pointers. Because each node holds hundreds of keys, the tree is astonishingly shallow — a B+tree over a billion rows is typically only 3–4 levels deep. Finding any key means reading about 4 nodes, not a billion rows.

1.Start at the root node (almost always cached in memory). Compare your key to its sorted keys; follow the one child pointer whose range contains it.
2.Land on an internal node. Compare again, follow the right pointer. Each step drops you one level and discards ~99% of what remained.
3.Arrive at a leaf node holding the exact key plus a pointer to the full row (or, for a clustered index, the row itself).
4.Follow the pointer, read the row, return it. Total work: a handful of node reads regardless of table size.

B+trees also keep their leaves linked in sorted order. That is why the same index that answers WHERE id = ? also makes WHERE id BETWEEN 100 AND 200, ORDER BY id, and WHERE id > 500 LIMIT 20 fast — you find the start, then walk the leaf chain. Range queries and sorting come free with a B-tree, which is exactly why it usually beats a hash index in practice.

3. The kinds of index, and when to reach for each

B-tree / B+tree

The default. Sorted, so it serves equality and ranges, sorting, and prefix matches. Your first choice unless you have a specific reason not to.

Great for: =, <, >, BETWEEN, ORDER BY, LIKE 'abc%'

Hash index

Hashes the key to a bucket — O(1) equality lookups, often faster than a B-tree for exact matches. But it is unsorted, so it cannot do ranges or ordering at all.

Great for: exact = only. In-memory tables, Redis.

Composite (multi-column)

One index over several columns, e.g. (user_id, created_at). Obeys the leftmost-prefix rule — column order is a design decision, not a detail.

Great for: queries filtering on the leading column(s).

Covering index

Includes every column the query needs, so the database answers entirely from the index and never touches the table — an "index-only scan," the fastest a read gets.

Great for: hot read paths where you control the exact query.

Clustered index

The table is the index — rows are physically stored in index order (InnoDB's primary key). One per table. Range scans on it are blisteringly fast; secondary indexes point back through it.

Great for: the primary key / most common range column.

Specialized

Partial (index only rows matching a condition), full-text (words in documents), GIN/GiST (arrays, JSON, geospatial), and LSM-trees (write-optimized, in Cassandra/RocksDB).

Great for: search, JSON, geo, and write-heavy stores.

4. The leftmost-prefix rule — the composite-index trap

A composite index on (user_id, created_at) is a catalog sorted first by user_id, and within each user, by created_at. That means it can serve some queries and not others:

Query filters on…Uses index (user_id, created_at)?
user_id = 7✅ Yes — leading column
user_id = 7 AND created_at > '2026-01-01'✅ Yes — full prefix, ideal
created_at > '2026-01-01' (no user_id)❌ No — skips the leading column, like searching a phone book by first name

Column order is everything. Put the column you always filter by first, and the one you range or sort by second. Getting this wrong is the single most common reason an index "exists but is not being used."

5. The hidden cost nobody mentions

Indexes are not a free lunch — they are a read-write trade, and beginners only see the read half:

Write amplification.Every INSERT, UPDATE, or DELETE must also update every index touching the changed columns, keeping each one sorted. Five indexes turn one logical write into six physical writes. On write-heavy tables this dominates.
Disk & memory.An index is a whole extra copy of its columns. Indexes can easily exceed the size of the table itself, and they compete with your data for precious RAM and cache.
Low cardinality is a trap.Indexing a column with few distinct values (a boolean like is_active) barely narrows anything — the planner often ignores it and scans anyway, because reading half the table through an index is slower than a straight scan. Index high-cardinality, selective columns.
The planner decides, not you.You create an index; the query optimizer chooses whether to use it based on estimated selectivity and its statistics. An index the planner never picks is pure cost. EXPLAIN / EXPLAIN ANALYZE is how you check.
Rule of thumb: index the columns you filter, join, and sort on frequently and selectively; do not reflexively index everything; and on an insert-heavy table, treat each extra index as a tax on ingestion you must justify.

6. Common mistakes & misconceptions

The misconceptionThe reality
"Indexes always make the database faster."They make reads faster and writes slower. On a write-heavy table, over-indexing is a net loss.
"Just add an index on every column I query."The planner may never use most of them; each is pure write + storage cost. A composite index often replaces several single-column ones.
"An index on (a, b) also covers queries on b alone."No — leftmost-prefix rule. It covers a and (a, b), not b by itself.
"More index = better; index the boolean flags too."Low-cardinality columns barely filter; the planner usually ignores the index and scans.
"I created the index, so my query is fast now."Only if the optimizer chooses it. Verify with EXPLAIN — an unused index is dead weight.

Real-World Engineering

🐘 PostgreSQL

Ships B-tree, hash, GiST, GIN, BRIN and more. GIN indexes power full-text search and jsonb queries; partial indexes (WHERE status='active') keep hot indexes tiny. Engineers live in EXPLAIN ANALYZE to confirm the planner actually chose their index.

🐬 MySQL / InnoDB

The primary key is a clustered index — rows are physically stored in PK order — so choosing a monotonic PK (not a random UUID) keeps inserts sequential and pages full. Every secondary index stores the PK as its row pointer, so a fat PK bloats every other index.

🪶 Cassandra / RocksDB

Use LSM-trees instead of B-trees: writes go to an in-memory sorted buffer and are flushed as immutable sorted files, later compacted. This flips the trade — writes become extremely cheap and sequential, at the cost of read amplification and background compaction.

🔎 Elasticsearch

Built entirely around the inverted index — a map from each term to the list of documents containing it — the same "sorted keyword → pointers" idea from the textbook analogy, specialized for full-text search at massive scale (we design one on Day 55).

REST API Design Best Practices
A Small Vocabulary for a Big System

You've spent two weeks making data survive, scale, and split across machines. But data that no one can reach is useless — mobile apps, web frontends, partner companies, and your own services all need a doorway in. Build that doorway carelessly and you get hundreds of one-off, bespoke commands nobody can remember. REST is the discipline that collapses all of it into a tiny, predictable vocabulary: things have addresses, and you act on them with the same handful of verbs. Master that idea and an API of any size stays learnable.

Visual Memory Anchor

📚🔤

Picture this

A vast library where every book sits at a permanent, printed address — /books/1984, /authors/orwell — and where you're handed a card with only four buttons: read it, add a new one, replace it, remove it. It doesn't matter whether you want a book, an author, or a borrowing record — the address names the thing, and the same four buttons do everything. And there's one strange rule: the librarian has total amnesia. Every time you approach the desk you must restate who you are and exactly what you want, because he remembers nothing from a moment ago. That amnesia is a feature — it means any librarian at any desk can serve you, so the library can open a hundred desks and never get confused. Nouns as addresses, a fixed set of verbs, and a clerk who forgets on purpose: that is REST.

The Story

Database Dave finally has his ledger sharded, replicated, and humming. Now the requests pour in: the mobile team wants balances, the web team wants transaction history, a partner bank wants to open accounts programmatically. So Dave does the obvious thing — he writes an endpoint for each need. /getUserById, /updateUserEmail, /deleteUserAccount, /getUserTransactions, /markUserActive, /getUserBalanceForMobile… Within a month there are three hundred endpoints, each a slightly different verb, each documented (or not) in a different way. New engineers can't guess a single one; every integration is a scavenger hunt.

A mentor looks at the mess and asks one question: "How many kinds of things does your system actually have?" Dave counts — users, accounts, transactions, cards. Maybe a dozen nouns. "So why do you have three hundred commands for a dozen things?" The insight lands: instead of inventing a verb for every action, Dave models each thing as a resource with an address — /users/42, /accounts/42/transactions — and reuses the verbs HTTP already gives him: GET to read, POST to create, PUT/PATCH to change, DELETE to remove. Three hundred bespoke endpoints collapse into a dozen predictable patterns anyone can guess. This is REST.

Then reality tests the design. A load balancer (Day 5) sends a partner's second request to a different server than the first — and Dave's server had been quietly remembering the login in memory, so it breaks. He learns the hard rule: the server must be stateless, carrying no memory between requests, so any replica can serve any call. A partner integration shatters overnight because Dave renamed a field — he learns versioning. A single GET /users tries to return ten million rows and melts a phone — he learns pagination. And a flaky network makes a partner retry a payment, creating it twice — he learns idempotency. Today is those hard-won rules, in order.

What should stay in your head
REST is an architectural style, not a protocol or a library. Its core trade is this: give up inventing custom verbs, and in return get a uniform interface that anyone can learn once and apply everywhere. Model your domain as resources (nouns) at stable URLs, act on them with HTTP's fixed set of verbs, and keep every request stateless so any server replica can serve it — which is exactly what lets you scale horizontally (Day 2) and load-balance freely (Day 5). Then lean on HTTP's own machinery: status codes to report outcomes, idempotency so retries are safe, and cacheability so GETs can be served from a CDN (Day 4). The whole payoff is predictability: a well-designed REST API is one you can guess.

How REST Actually Works

1. Resources are nouns; the URL is their address

The single most important shift is this: put the noun in the URL and the verb in the HTTP method. A URL identifies a thing (or a collection of things), never an action. GET /users/42 — not /getUser?id=42. Collections are plural nouns (/users), a single item is that collection plus an id (/users/42), and relationships nest (/users/42/orders = "the orders belonging to user 42"). When the URL names things and the method names the action, an API of a thousand routes still follows five or six patterns you can guess without reading the docs.

2. The uniform interface: one small set of verbs

REST rides on HTTP's existing methods. Three properties decide how you use each one, and they matter enormously the moment the network gets flaky:

  • Safe — does it change anything? (A safe method only reads.)
  • Idempotent — does doing it twice land you in the same state as doing it once? (Critical for safe retries.)
  • Cacheable — can the response be reused without re-asking the server? (Ties straight back to CDNs, Day 4.)
MethodMeansSafe?Idempotent?Cacheable?
GETRead a resourceYesYesYes
POSTCreate a new resource under a collectionNoNoNo
PUTReplace a resource entirely (at a known id)NoYesNo
PATCHModify part of a resourceNoUsually noNo
DELETERemove a resourceNoYesNo

The idempotency column is not trivia. Networks lose responses all the time: your request succeeded, but the "200 OK" never made it back, so the client retries. If the method is idempotent (PUT /users/42, DELETE /orders/9), the retry is harmless. If it's POST /payments — not idempotent — the retry charges the customer twice. That's why real payment APIs add an idempotency key to make POST safe to retry (we'll see Stripe do exactly this below).

3. Statelessness: the server remembers nothing

Every REST request must carry everything the server needs to fulfill it — identity, auth token, parameters. The server keeps no session memory between requests. This sounds like a limitation; it's actually the property that makes the whole thing scale. Because request #2 doesn't depend on server memory from request #1, a load balancer (Day 5) can send each request to any replica, you can add or kill servers at will (Day 2's horizontal scaling), and a crashed server takes no session state down with it. State still exists — but it lives in the database or a shared cache (Redis, Day 7), not in one server's RAM. "Stateless server, stateful store" is the pattern.

4. Status codes: let HTTP report the outcome

Don't invent your own success/failure scheme inside a 200 OK body. Use the codes HTTP already defines so any client, proxy, and cache understands the result:

2xx — it worked.200 OK (read/updated), 201 Created (new resource made — return its URL), 204 No Content (worked, nothing to send back, e.g. a DELETE).
3xx — go elsewhere.301/302 redirects, 304 Not Modified (your cached copy is still good — the backbone of HTTP caching).
4xx — you (the client) made a mistake.400 bad request, 401 not authenticated, 403 authenticated but not allowed, 404 no such resource, 409 conflict, 429 too many requests (rate limited — Day 24).
5xx — we (the server) broke.500 internal error, 503 unavailable. The client/server split matters: 4xx says "don't retry, fix your request"; 5xx says "not your fault, a retry might work."

5. The Richardson Maturity Model (how "RESTful" is it, really?)

REST isn't binary. Leonard Richardson described it as a ladder: Level 0 is one URL, one verb (basically RPC over HTTP). Level 1 introduces resources (many URLs). Level 2 uses HTTP verbs and status codes properly — this is where the vast majority of good, pragmatic APIs live and stop. Level 3 adds HATEOAS: responses include links telling the client what it can do next, so the client discovers the API instead of hard-coding URLs. Level 3 is elegant and rarely reached in practice — worth knowing exists, not worth agonizing over. Being honest that "Level 2 is the real target" saves teams a lot of wasted purity debates.

The Design Decisions (Where APIs Live or Die)

Versioning

Clients you don't control depend on your response shape, so you can't change it freely. Version from day one — in the path (/v1/users, blunt but obvious) or a header (cleaner URLs, harder to test). Never make a breaking change to a live version; add a new one and deprecate the old on a schedule.

Rule: additive changes are safe; renaming or removing fields is not.

Pagination

Never return an unbounded list. Offset (?page=3&limit=50) is simple but drifts when rows are inserted and gets slow deep in the list. Cursor (?after=token) is stable and fast at any depth — the right default for large, changing datasets. (Notice cursors are just the shard/index keys from Days 11–13 surfacing in the API.)

Great for feeds/timelines: cursor. Great for a small admin table: offset.

Filtering & sorting

Keep the resource URL clean and push refinement into the query string: /orders?status=paid&sort=-created_at. Don't bake filters into new endpoints (/getPaidOrdersSortedByDate) — that's the endpoint explosion sneaking back in.

Query params refine a collection; they don't create a new resource.

Errors & idempotency keys

Return a consistent, machine-readable error body (a stable code, a human message) alongside the right status. For non-idempotent creates, accept an idempotency key header so a retried POST is de-duplicated instead of charging twice.

One error shape everywhere beats a bespoke error per endpoint.

Naming conventions that keep an API guessable

Small, boring consistency is what makes a large API feel small: plural nouns for collections (/users, not /user); no verbs in paths (the method is the verb); lowercase, hyphenated segments; nest relationships only one level deep (/users/42/orders, but fetch a deep order by its own id, /orders/9, not /users/42/orders/9/items/3). The test of a good REST API: a competent engineer who has seen three of your endpoints can correctly guess the fourth.

What REST Doesn't Solve — and Why Days 16–17 Exist

REST's greatest strength — one uniform, resource-shaped interface — is also the source of its sharpest limits. Because the server decides the shape of each resource, the client is stuck with it:

Over-fetching.A phone screen needs a user's name and avatar, but GET /users/42 returns the full 40-field record. You ship bytes nobody uses — painful on mobile networks.
Under-fetching & the N+1 problem.To render a feed you fetch /posts, then loop and fetch /users/{id} for each author and /posts/{id}/comments for each post — dozens of round trips for one screen. A chatty client on a high-latency link feels every one.
One-size-fits-all resources.Different clients (watch, phone, web, TV) want different slices of the same data, but they all get the same fixed resource shape.

These aren't bugs in REST — they're the cost of its uniformity. GraphQL (Day 16) answers by letting the client ask for exactly the fields it wants in one request. gRPC (Day 17) answers with a compact binary, contract-first protocol for fast service-to-service calls. REST remains the right default for public, cacheable, resource-shaped APIs — but knowing precisely where it strains is what tells you when to reach for the alternatives.

Common Mistakes & Misconceptions

The misconceptionThe reality
"Put the action in the URL — /createUser, /deleteUser."That's RPC, not REST, and it brings back the endpoint explosion. The URL names the noun (/users); the HTTP method is the verb (POST/DELETE).
"Just use POST for everything and return 200."You throw away idempotency, caching, and meaningful errors. Use GET/PUT/DELETE where they fit so retries are safe and responses are cacheable, and return honest status codes.
"The server can keep the user's session in memory."That breaks the instant a load balancer routes the next request to another replica, and dies with the server. Keep servers stateless; put shared state in a store (DB/Redis).
"PUT and PATCH are the same thing."PUT replaces the whole resource (and is idempotent); PATCH modifies part of it (often not idempotent). Sending a partial body as a PUT can wipe the fields you omitted.
"Retrying a failed request is always safe."Only for idempotent methods. Retrying a POST /payments whose response was lost can double-charge. Add an idempotency key to make creates safe to retry.
"We'll add versioning later if we need it."Once external clients depend on your shapes, an unversioned breaking change takes them all down at once. Version from the first release; make only additive changes within a version.
"REST means JSON."REST is an architectural style independent of format — it predates JSON's dominance and works with XML, Protobuf, anything. JSON is just the common convention, not the definition.

Active Recall

❓ A partner calls POST /payments to charge a customer. Your server succeeds and saves the payment, but the network drops the 201 Created response on the way back. The partner's client, seeing no reply, retries the exact same request. What happens, why, and how should the API have been designed to prevent it? (click to reveal)
The customer gets charged twice. POST is not idempotent — each call creates a new resource — so the server has no way to know the second request is a retry of the first rather than a genuine second payment. The fix is an idempotency key: the client generates a unique key (say a UUID) and sends it as a header with the request; the server records "I've already processed key X → here's the result I returned," so any retry with the same key returns the original result instead of creating a second charge. This turns a non-idempotent POST into a safely retryable one — which is exactly why every serious payments API (Stripe, PayPal, Adyen) requires idempotency keys on creates. The deeper lesson: on an unreliable network, "did it succeed?" is genuinely unknowable to the caller, so safe retries must be designed in, not assumed.

Real-World Engineering

💳 Stripe

The reference-quality REST API. Clean plural-noun resources (/v1/charges, /v1/customers), date-based versioning pinned per account so old integrations never break, and mandatory idempotency keys on every create so a retried charge is de-duplicated, not doubled.

🐙 GitHub

A huge, guessable REST surface (/repos/{owner}/{repo}/issues), cursor-style pagination via the Link header, ETags with 304 Not Modified for cheap conditional reads, and one of the closest things to HATEOAS in the wild — responses embed URLs to related resources.

🐦 X / Twitter

At timeline scale, offset pagination collapses, so their APIs are built on cursors — opaque tokens that page stably through a feed that's changing under you. A live demonstration that pagination strategy is a scaling decision, not a formatting one.

☁️ Google & AWS

Both publish detailed API design guides that standardize REST across thousands of engineers: consistent resource naming, standard methods, uniform error shapes, and explicit versioning — proof that at scale, the discipline matters more than any single clever endpoint.

Flashcards (Spaced Repetition)

Q: In one line, what is REST's core trade-off?
A: You give up inventing custom verbs and, in return, get a uniform interface — resources (nouns) at stable URLs acted on by a fixed set of HTTP verbs — that anyone can learn once and apply everywhere, keeping even a huge API predictable.
Q: Why must a REST server be stateless, and where does state go instead?
A: So any replica can serve any request — enabling horizontal scaling (Day 2) and load balancing (Day 5), and surviving server crashes. State doesn't vanish; it moves out of a single server's memory into a shared store (database or Redis, Day 7).
Q: What does idempotent mean, and which HTTP methods are idempotent?
A: Doing it twice leaves the system in the same state as doing it once. GET, PUT, and DELETE are idempotent; POST and (usually) PATCH are not. Idempotency is what makes a retry safe after a lost response.
Q: How do you make a non-idempotent POST (like creating a payment) safe to retry?
A: Use an idempotency key — the client sends a unique key with the request; the server records the result for that key and returns the same result on any retry, instead of creating a second resource. (Stripe requires this on all creates.)
Q: Offset vs cursor pagination — when do you use each?
A: Offset (?page&limit) is simple but drifts when rows change and slows down deep in the list — fine for small, stable tables. Cursor (?after=token) is stable and fast at any depth — the right default for large, changing feeds and timelines.

Memory Test — Answer Without Looking Up

  • Rewrite these RPC endpoints the RESTful way: /getUser?id=42, /updateUserEmail, /deleteUser, /listUserOrders.
  • Explain statelessness in one sentence, and name two things it makes possible that a stateful server can't.
  • For each — GET, POST, PUT, PATCH, DELETE — say whether it's safe, idempotent, and cacheable, and give the one that most surprises people.
  • A retried POST /orders creates a duplicate order. Diagnose why and give the concrete fix.
  • When would you choose cursor pagination over offset pagination, and why does offset break at scale?
  • Name the four status-code families (2xx–5xx) and the practical difference between a 4xx and a 5xx for a client deciding whether to retry.
  • Give two concrete situations where REST strains, and which of GraphQL or gRPC you'd reach for in each.
If you only remember one thing: REST trades away custom verbs to buy predictability — things get addresses (nouns in the URL), a fixed handful of verbs act on them (the method is the verb), and the server remembers nothing so any replica can serve any request. Everything else — honest status codes, idempotent retries, versioning, pagination — is HTTP's own machinery doing the heavy lifting. Build it so a stranger can guess your fourth endpoint after seeing three, and you've built a good API. Where that uniform shape starts to chafe is precisely where tomorrow's GraphQL and gRPC begin.

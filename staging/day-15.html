REST API Design
Build a Front Door Anyone Can Use

For two weeks you've built the machinery behind the counter — replication, sharding, the ring. Today we design the counter itself: the API that the outside world actually talks to. REST is a surprisingly small idea — expose your system as a handful of nouns (resources), let callers act on them with a fixed set of verbs (HTTP methods), and keep the server stateless so it remembers nothing between requests. Get it right and thousands of developers use your API without ever reading a manual. Get it wrong and every new feature becomes a negotiation.

Picture this

Your API is a restaurant. The menu lists the things you can order — the pasta, the steak, table 7 — these are nouns, and every one has a number you point at (that's your URL: /dishes/pasta). You never shout "executePastaCooking()" into the kitchen; you use one of a tiny, universal set of gestures — bring me (GET), add this (POST), replace this (PUT), take it away (DELETE). The waiter carries every detail on a written ticket, because the kitchen is stateless: it remembers nothing about your last visit, so each ticket must be self-contained. What lands on your table is a plating — a JSON representation of the abstract menu item. And the waiter always answers in the same standard phrases: "coming right up" (200), "never heard of that dish" (404), "kitchen's on fire" (500). Hold that picture — menu of nouns, a handful of standard gestures, a forgetful kitchen, standard replies — and you're holding REST.

The Story

Database Dave finally had a storage layer he trusted — replicated, sharded, sitting on a tidy hash ring. But nobody in the world could use it yet. The mobile team, the web team, a partner company, and a batch job all needed to read and write the same data, and each one was reaching straight into the database with its own hand-rolled queries. Every schema tweak broke four things at once. Dave needed a single, stable front door — one place everyone talked to — so the storage behind it could change without the whole world noticing.

His first attempt was a mess. He shipped endpoints as they came to mind: /getUser?id=42, /createNewUserAccount, /updateUserThenSendEmail, /user/42/delete. Six months later there were 400 of them, no two shaped alike, and onboarding a new developer meant handing them a 30-page document that was already out of date. "Every endpoint is a little snowflake," a teammate sighed. "I have to learn each one individually." That's the tell of an API with no system behind it — the surface area grows faster than anyone can memorize.

A senior engineer rebuilt it around one discipline. "Stop inventing verbs," she said. "You have things — users, orders, payments. Give each kind of thing one address as a collection (/users) and one per item (/users/42). Then use the four verbs HTTP already gives you to act on them." Suddenly the 400 snowflakes collapsed into a pattern anyone could guess: GET /users lists, POST /users creates, GET /users/42 reads, PUT /users/42 replaces, DELETE /users/42 removes. New developers stopped reading the manual — they could predict the API. That predictability is the entire point of REST, and it's what the rest of this lesson makes concrete.

What should stay in your head
REST turns your system into a small set of resources (nouns with addresses) acted on by a uniform set of verbs (the HTTP methods), over a stateless server that keeps no per-client memory between requests. The URL names what (the resource); the method says what to do to it; the status code reports how it went; the body is a representation (usually JSON) of the resource's state. Do this consistently and callers can guess your API instead of memorizing it — that predictability is the whole payoff.

How REST Actually Works


        
REST — REpresentational State Transfer — was
        described by Roy Fielding in 2000 as a set of constraints, not a library. It's not "an API that
        returns JSON"; it's a style built on a few rules that, followed together, make an API predictable and
        scalable. Four of those rules do most of the work.


        
1 — Resources are nouns with addresses

        
The unit of a REST API is the resource: a thing your system knows about — a user, an
        order, a comment. Each resource type gets a collection URL and each instance gets its
        own. Use plural nouns, and let hierarchy express ownership:

        

          
/usersthe collection of all users

          
/users/42one specific user — the item

          
/users/42/ordersthe orders belonging to user 42 (a sub-collection)

          
/users/42/orders/7one specific order of that user

        

        
Notice there are no verbs in the path. /getUser,
        /createOrder, /users/42/delete are all antipatterns — the action belongs in the
        HTTP method, not the URL. The URL only ever names what, never what to do.


        
2 — A uniform, fixed set of verbs

        
HTTP already gives you the verbs. You don't invent them — you reuse the same handful for every
        resource, which is exactly why callers can guess your API:

        

          
MethodOn a collection /usersOn an item /users/42Safe?Idempotent?

          
GETlist usersread user 42✅ yes✅ yes

          
POSTcreate a new user—❌ no❌ no

          
PUT—replace user 42 entirely❌ no✅ yes

          
PATCH—update part of user 42❌ no⚠️ usually not

          
DELETE—remove user 42❌ no✅ yes

        

        
Safe means the request doesn't change server state — you can call it a million times
        and read-only stays read-only. Idempotent means calling it once or many times leaves the
        server in the same final state. These aren't trivia: they're what let networks retry safely. If a
        PUT or DELETE times out, a client (or load balancer, or the phone on a flaky
        connection) can just resend it. Retry a POST blindly, though, and you might create the same
        order twice — which is exactly why payment APIs bolt on idempotency keys (Day 30's topic).


        
3 — Statelessness

        
Each request must carry everything the server needs to handle it — who you are (a token), what
        you want, any parameters. The server keeps no session memory of you between calls. This
        feels wasteful, but it's the property that makes the front door scale: because no server holds
        "your" state, any of a thousand identical servers can handle your next request. That's why the load
        balancer from Day 5 can spray requests across the fleet freely — there's no "sticky" server that alone
        knows your context. Statelessness is what turns one API server into a horizontally scalable herd.


        
4 — Representations, not the thing itself

        
You never get "the user" — you get a representation of it, typically JSON, that
        captures its current state. The same resource could be represented as JSON, XML, or CSV; the client says
        which it wants via the Accept header (content negotiation). The resource is the abstract
        concept; what crosses the wire is a snapshot of its state — which is the literal meaning of
        "Representational State Transfer."

      


      

        
Status Codes — The Server's Standard Replies

        
The status code is how the server tells the caller what happened, in a language every HTTP
        client on Earth already understands. Using them precisely is a huge part of a good API — returning
        200 OK with {"error": "..."} in the body is a classic sin that breaks every
        client's error handling. Learn the families first, then a few specific codes.

        

          
CodeFamilyMeans

          
200 OK2xx — successRequest succeeded; the response has the data (GET/PUT/PATCH).

          
201 Created2xx — successA new resource was created (POST); return its URL in a Location header.

          
204 No Content2xx — successSucceeded, nothing to return (common for DELETE).

          
301 / 302 / 3043xx — redirectMoved, or "use your cached copy" (304 Not Modified powers caching).

          
400 Bad Request4xx — client's faultMalformed request — bad JSON, missing field. The caller should fix it.

          
401 / 4034xx — client's fault401 = not authenticated (who are you?); 403 = authenticated but not allowed (Day 27–28).

          
404 Not Found4xx — client's faultThat resource doesn't exist.

          
409 Conflict4xx — client's faultRequest clashes with current state — e.g. duplicate, or a stale update.

          
422 Unprocessable4xx — client's faultSyntax is fine but the data fails validation (e.g. email already taken).

          
429 Too Many Requests4xx — client's faultYou're rate-limited — slow down (Day 26).

          
500 / 502 / 5035xx — server's faultThe server broke, a gateway got a bad response, or the service is overloaded/down.

        

        
The single most useful rule: 4xx means the caller made a mistake; 5xx means you did.
        That one split tells a client whether retrying could ever help (5xx: maybe; 4xx: not until you change the
        request) and tells your on-call engineer whose problem it is at 3 a.m.

      


      

        
Design Rules That Separate Good APIs from Bad

        
Beyond the four constraints, a handful of concrete habits are what make an API pleasant to live with
        for years:

        

          
Version from day one.Put /v1/ in the path (or a header). The moment real clients depend on you, you can't change a response shape without breaking them — a version lets you ship /v2/ while /v1/ keeps working.

          
Page, filter, and sort with query params.GET /users?role=admin&sort=created&page=2&limit=50. Never return an unbounded list — a collection with 10 million rows will take the server down. The path identifies the resource; the query string refines the view of it.

          
Return the created resource.On 201, send back the new object (with its server-assigned id) and a Location header — so the client doesn't have to make a second call to learn what it just made.

          
Make errors machine-readable and consistent.Every error, everywhere, in the same shape: a stable code, a human message, and enough detail to act on. Clients build on the shape, not the prose.

          
Be consistent about names.Pick snake_case or camelCase and never mix. Plural collections everywhere. If created_at is a timestamp here, it's a timestamp everywhere. Consistency is what lets people guess.

          
Design for idempotent retries.Assume every request might arrive twice (networks retry). Lean on naturally-idempotent verbs where you can, and for POST, accept an idempotency key so a retried "create payment" doesn't charge twice.

        

      


      

        
REST vs the Alternatives (It's Not the Only Front Door)

        

          

            
REST over HTTP

            
Resources + HTTP verbs + status codes, as above. Ubiquitous, cacheable, human-readable, works with
            any HTTP client and a browser. The safe default for public and internal APIs alike.

            
Great for: public APIs, CRUD-shaped services. Used by: Stripe, GitHub, Twilio.

          

          

            
GraphQL (Day 16)

            
One endpoint; the client writes a query describing exactly the fields it wants across
            many resources in a single round trip. Kills over-fetching and under-fetching, at the cost of caching
            and complexity moving onto you.

            
Great for: rich client UIs, many-shaped reads. Used by: GitHub (v4), Shopify, Meta.

          

          

            
gRPC / Protobuf (Day 17)

            
A binary, contract-first RPC framework over HTTP/2. Tiny, fast payloads and streaming — but not
            human-readable and awkward from a browser. This is the "verbs, not resources" model, done well.

            
Great for: internal service-to-service calls at scale. Used by: Google, Netflix internals.

          

          

            
Plain RPC (the antipattern-adjacent)

            
Endpoints that are named actions — /doThing. It's what Dave built first. Sometimes
            genuinely right (truly action-oriented operations), but as a whole-API style it loses REST's
            predictability and caching.

            
Great for: a few genuine actions. Risky as: your entire API surface.

          

        

        
The through-line: all four are just answers to "how does the outside world talk to my system?" REST
        wins on predictability and caching; GraphQL on flexible reads; gRPC on
        raw internal performance. Most real companies run all three at once — REST at the public
        edge, gRPC between internal services, GraphQL for a demanding front-end. Knowing why you'd pick
        each is exactly the kind of tradeoff a principal engineer is expected to reason about out loud.

Common Mistakes & Misconceptions

        

          
The misconceptionThe reality

          
"Any API that returns JSON is RESTful."REST is a set of constraints — resource-based URLs, uniform verbs, statelessness, proper status codes. An endpoint like POST /getUserData returning JSON is RPC wearing a JSON costume, not REST.

          
"Put the action in the URL: /users/42/delete."The verb belongs in the HTTP method. DELETE /users/42. URLs name things; methods name actions. Verbs-in-URLs multiply your endpoints and kill predictability.

          
"Return 200 OK and put errors in the body."Then every client has to parse the body to know if the call failed, and caches/monitoring get it wrong. Use the status code: 4xx for the caller's mistake, 5xx for yours.

          
"PUT and PATCH are the same thing."PUT replaces the whole resource (send the full object); PATCH updates part of it (send only the changed fields). Send a partial body to PUT and you may wipe the omitted fields.

          
"Idempotent means 'read-only'."No — safe means read-only. Idempotent means repeating it lands in the same final state. DELETE isn't safe (it changes things) but it is idempotent (deleting twice still = gone).

          
"I'll add versioning later when I need it."By the time you need it, real clients depend on your shape and you can't change it without breaking them. Ship /v1/ before your first external caller, not after.

          
"Statelessness means I can't have logins."You can — the client just sends its identity (a token) on every request instead of the server holding a session. The state moves to the token, not away entirely.

        

      


      

        
Active Recall

        

          
❓ A mobile client sends POST /payments to charge a card, but the response times out on a flaky connection — the app never learns whether it worked. If it just retries the POST, what could go wrong, and how do REST's method semantics plus one small addition fix it? (click to reveal)

          
Because POST is not idempotent, a blind retry can create a second payment — the customer gets charged twice. The timeout means the client can't tell if the first request reached the server, committed, and only the response was lost. The general fix is to make the operation idempotent: the client generates an idempotency key (a unique ID for this charge attempt) and sends it as a header. The server records the key with the result; if the same key arrives again, it returns the original result instead of charging again. This is exactly how Stripe's API works, and it's why understanding safe vs. idempotent verbs (this lesson) is the setup for the full idempotency treatment on Day 30. Meanwhile, if the same client had done PUT /users/42 and it timed out, retrying would be perfectly safe — PUT is idempotent by design, so the second call just re-sets the same state.

        

      


      

        
Real-World Engineering

        

          
💳 Stripe
Often held up as the gold standard for REST: clean plural resources (/v1/charges, /v1/customers), correct status codes, consistent error objects, cursor pagination, and first-class idempotency keys on writes so a retried payment never double-charges. Their API design is a big part of why developers love it.

          
🐙 GitHub
Runs a large, well-documented REST API (GET /repos/{owner}/{repo}/issues) — and then shipped GraphQL v4 alongside it because clients kept over-fetching across many REST calls to build a single screen. A living example of "REST at the edge, GraphQL for rich reads."

          
📞 Twilio
Exposes telephony as resources — POST /Messages to send an SMS, GET /Calls to list calls. Turning "send a text" into "create a Message resource" is the perfect illustration of the noun-not-verb mindset: even an action becomes a resource you create.

          
🔤 Google & the industry
Google publishes a widely-followed API Design Guide standardizing resource naming, standard methods, and status codes across hundreds of services — proof that at scale, REST conventions aren't style preferences, they're how thousands of engineers stay consistent without meeting.

        

      


      

        
Flashcards (Spaced Repetition)

        

          
Q: In one sentence, what is REST?

          
A: A style that exposes a system as resources (nouns with URLs) acted on by a fixed set of HTTP verbs, over a stateless server, with the outcome reported via status codes — so callers can predict the API instead of memorizing it.

          
        

        

          
Q: What's the difference between "safe" and "idempotent"?

          
A: Safe = doesn't change server state (GET). Idempotent = calling it once or many times leaves the same final state (GET, PUT, DELETE). DELETE is idempotent but not safe. POST is neither.

          
        

        

          
Q: PUT vs PATCH?

          
A: PUT replaces the entire resource — send the full object. PATCH updates part of it — send only the fields you're changing. Sending a partial body to PUT can blank out the fields you omitted.

          
        

        

          
Q: What does a 4xx status code mean vs a 5xx?

          
A: 4xx = the caller's fault (bad request, unauthorized, not found) — retrying without changing the request won't help. 5xx = the server's fault (crashed, overloaded, bad gateway) — retrying might succeed later.

          
        

        

          
Q: Why does statelessness make an API scale?

          
A: Because no server holds "your" session, any server in the fleet can handle your next request — so a load balancer can spread traffic across a horizontally scalable herd. The client carries its own context (a token) on every call.

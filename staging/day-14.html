Consistent Hashing
How to Add a Machine Without Moving Everything

Yesterday's sharding left us with a demon: naive hash(key) mod N is beautifully balanced right up until you add or lose a machine — and then the modulus changes for almost every key, so roughly 80% of your data has to move at once. That single flaw makes elastic, always-on systems impossible. Consistent hashing is the deceptively simple idea that fixes it: bend the number line into a ring, place both the keys and the machines on it, and let each key belong to the next machine clockwise. Add a machine and it steals keys from just one neighbor — on average only 1/N of the data moves. It's the quiet circle underneath nearly every distributed cache and database you'll ever touch.

Picture this

Picture a giant clock face, but instead of 12 hours it has billions of tiny tick marks running all the way around. You scatter your machines onto the rim like numbers on the clock, and you scatter your keys onto the rim too. The rule is a single sentence: each key belongs to the first machine you hit going clockwise from where the key landed. That's the whole idea. Now watch the magic — when you add a new machine, you drop it somewhere on the rim, and it only picks up the keys sitting in the small arc just behind it, stolen from the one neighbor it now stands in front of. Every other key on the entire clock stays exactly where it was. Remember the clock, remember "next one clockwise," and you remember why adding a server no longer means rebuilding the world.

The Story

Database Dave ended yesterday sharding his data with hash(user_id) mod 4 across four machines. Load was perfectly even and he went home happy. Then his app went viral and four machines couldn't keep up, so — reasonably — he added a fifth and changed the code to mod 5. That night the cluster nearly fell over. The modulus had changed for almost every key: a user who used to live on machine 2 now hashed to machine 4, another jumped from 0 to 3, and on and on for roughly 80% of all users. The system spent hours frantically copying data between machines while still trying to serve live traffic. Dave had discovered that mod N doesn't add a machine — it reshuffles the entire deck.

A colleague draws him a picture: a circle. "Stop thinking of your machines as slots 0 through 4," she says. "Put them on a ring. Hash each machine's name to a point on the ring, and hash each key to a point too. A key belongs to the first machine clockwise from it." Dave squints. "Now add a machine," she says, dropping a new point onto the circle. Only the keys in the little arc behind the new machine change owners — everything else on the ring doesn't move an inch. "Adding a machine steals from one neighbor, not from everyone. On average you move one-Nth of the data, not four-fifths of it." This is consistent hashing.

Dave tries it and hits one snag. With only five machines placed randomly on the ring, the arcs come out uneven — one machine sits in front of a huge arc and gets slammed while another gets a sliver. He's traded the rebalancing demon for a balance demon. The fix turns out to be wonderfully cheap: instead of placing each machine once, place it a hundred times under names like M1#1, M1#2, … M1#100 — virtual nodes. Now each machine owns a hundred tiny scattered arcs that average out smoothly, and when a machine dies its hundred little arcs spill evenly onto everyone else instead of dumping onto a single unlucky neighbor. Today is about that ring, that clockwise rule, and the virtual-node trick that makes it actually balanced.

What should stay in your head
Plain hash(key) mod N ties every key's home to the count of machines, so changing the count moves almost everything. Consistent hashing ties a key's home to a position on a ring instead: hash keys and machines onto the same circular space, and each key goes to the next machine clockwise. Adding or removing a machine only disturbs the one arc next to it, so on average just 1/N of keys move — proportional, not catastrophic. The one weakness — uneven arcs causing unbalanced load — is solved by virtual nodes: place each machine at many points on the ring so its share averages out and failures spread evenly.

How Consistent Hashing Actually Works

Bending the number line into a ring

A hash function turns any input into a number in a fixed range — say 0 to 232−1. Normally we'd take that number mod N to pick a machine. Consistent hashing does something different: imagine that whole range wrapped into a circle, so the largest value sits right next to 0. Now hash each machine's identifier (its name or IP) to a point on that circle, and hash each key to a point too. Both live in the same circular space. To find a key's home, start at the key's position and walk clockwise until you bump into the first machine. That machine owns the key. In practice you don't walk — you keep the machine positions in a sorted structure and binary-search for "the smallest machine position ≥ the key's position" (wrapping around to the first machine if you fall off the end). That's an O(log N) lookup, no network call required.

Why adding a machine only moves 1/N of the keys

This is the whole payoff, so it's worth seeing clearly. Each machine on the ring is responsible for the arc that ends at it — every key between the previous machine and this one, going clockwise. When you drop a new machine onto the ring, it lands inside exactly one existing arc and splits it in two. The new machine takes over the first half of that arc; the machine that used to own the whole arc keeps the second half. Every other arc on the ring is untouched. So the only keys that change owners are the ones in that one split arc — on average 1/(N+1) of all keys. Compare that to mod N, where changing N re-computes the destination for essentially every key. Removal is the mirror image: when a machine leaves, its arc simply merges into the next machine clockwise, and only that machine's keys move. Failures and additions become local events, not global reshuffles.

1. Place machines on the ring.Hash each machine's name/IP to a point on the circle. Keep the points in a sorted list.
2. Place a key.Hash the key to a point on the same circle.
3. Walk clockwise.The first machine at or after the key's point owns it (wrap to the first machine if you pass the end). Binary search, O(log N).
4. Add a machine.Insert one new point. It splits a single arc — only that arc's keys move to it. Everything else stays put.
5. Remove a machine.Delete its point. Its arc merges into the next machine clockwise — only its keys move.

The catch: raw rings are lumpy

If you place each machine at exactly one random point, the arcs between them come out uneven by pure chance — some machines end up in front of a wide arc and get far more than their fair share, others get a sliver. With only a handful of machines this imbalance can be severe (easily 2× off). And there's a second problem: when one machine dies, its entire arc dumps onto a single neighbor, instantly doubling that neighbor's load — exactly the kind of cascading overload that takes clusters down. Both problems have the same elegant fix, and it's the second half of today's lesson.

Virtual Nodes: Making the Ring Fair

Instead of placing each physical machine at one point on the ring, place it at many — say 100 or 200 — each derived from a different label like hash("M1#0"), hash("M1#1"), and so on. These are virtual nodes (or "vnodes"). A physical machine now owns a hundred small, scattered arcs instead of one big one. Two wonderful things follow:

Smoother balance

A hundred random arcs per machine average out by the law of large numbers, so every machine ends up with close to its fair 1/N share. More vnodes → tighter balance (the sim above makes this visceral: crank the slider and the bars flatten).

Rule of thumb: 100–200 vnodes/machine keeps imbalance to a few percent.

Graceful failure

When a machine dies, its hundred little arcs spill onto many different neighbors — a few percent each — instead of doubling one unlucky machine. Load redistributes evenly and no single node gets a sudden spike.

This is the real reason production systems always use vnodes, not just balance.

Heterogeneous hardware

Got a machine with twice the RAM? Give it twice the vnodes. Its share of the ring — and so its share of the data — scales with how many points it holds. Capacity becomes a simple dial.

How Cassandra/Dynamo weight bigger nodes without special-casing the algorithm.

The cost is modest: more points to store and sort, and a slightly larger routing table. But it's cheap memory in exchange for even load and painless failure, which is why every real consistent hashing deployment uses virtual nodes. When someone says "consistent hashing," they almost always mean "consistent hashing with vnodes."

Limits & Refinements (What It Doesn't Fix)

Consistent hashing solves rebalancing cost and, with vnodes, data distribution. It does not, by itself, solve everything:

Hot keys still hurt.Just like yesterday: one wildly popular key hashes to one point and lands entirely on one machine. Consistent hashing spreads distinct keys, not the traffic of a single key. Fixes are still app-level — replicate/cache the hot key (Day 6).
It's usually paired with replication.Real systems store each key on the next R machines clockwise, not just the first — so a key survives a node loss. This is how Dynamo-style "N replicas on the ring" works, and it dovetails with the quorum math from Day 10.
Randomness can still spike load.Even with vnodes, load is only statistically even. Consistent hashing with bounded loads adds a cap: if a machine is already at, say, 1.25× the average, keys skip past it to the next one. Google uses this to guarantee no node is overloaded.
Sometimes a different algorithm fits better.Rendezvous (HRW) hashing and jump consistent hashing achieve the same "minimal movement" goal with less memory and no explicit ring — good when you have a fixed, known set of buckets and want tiny, fast lookups.

Common Mistakes & Misconceptions

The misconceptionThe reality
"Consistent hashing means every machine always gets exactly equal load."No — it minimizes data movement on resize. Balance comes from virtual nodes. A raw ring with one point per machine is often badly unbalanced.
"It makes hot keys go away."It spreads distinct keys evenly, but a single hot key still hashes to one point and lands on one machine. Hot keys are still an app-level problem (replicate/cache them).
"Adding a node re-hashes and moves everything, like mod N."The opposite — that's the entire point. Adding a node splits one arc, moving only ~1/N of keys. Everything else stays exactly where it is.
"Virtual nodes are an optimization you can skip."In production you never skip them. Without vnodes, load is lumpy and a node failure dumps its whole arc onto one neighbor, risking a cascade.
"The ring itself stores the data."The ring is just a routing map — a sorted set of machine positions used to answer "who owns this key?" The data lives on the machines; the ring only decides which one.
"More machines automatically means each request is faster."Consistent hashing balances storage and load, not per-request latency. It's about elasticity and even distribution, not making a single lookup quicker.

Active Recall

❓ Dave switches from hash(user_id) mod N to consistent hashing and adds a 6th machine. (a) Roughly what fraction of keys move now, versus what mod N would have moved? (b) After the switch he notices one machine is at 2× the load of another — what went wrong and how does he fix it? (click to reveal)
(a) With consistent hashing, adding the 6th machine splits a single arc, so only about 1/6 (~17%) of keys move — and they all come from one neighbor. Plain mod N (5→6) would have re-computed the destination for roughly 5/6 (~83%) of all keys, a near-total reshuffle. (b) The imbalance is the raw-ring lumpiness problem: with only a few points on the ring, arcs come out uneven by chance, so one machine sits in front of a much wider arc. The fix is virtual nodes — place each machine at ~100–200 points instead of one, so its many small arcs average out to a near-equal share, and a failure spreads its load across many neighbors instead of doubling one.

Real-World Engineering

🛒 Amazon DynamoDB / Dynamo

The original Dynamo paper popularized consistent hashing for a production key-value store: keys map onto a ring, each is replicated to the next N nodes clockwise, and virtual nodes ("tokens") balance load and let nodes join/leave with minimal movement. This design underpins DynamoDB.

🪶 Apache Cassandra

Cassandra is a ring, literally — nodes own token ranges on a consistent-hash ring, use vnodes (default 16, historically 256) to spread load, and replicate each key to the next R nodes. Adding a node streams over only the ranges it now owns, no cluster-wide reshuffle.

💬 Discord

Discord uses consistent hashing to route millions of concurrent users to the right session/guild servers, so scaling the server pool up and down only remaps a small slice of connections instead of stampeding everyone onto new nodes.

🌐 CDNs & caches (Akamai, memcached)

CDN request routing and sharded memcached clients use consistent hashing so that adding or losing a cache node invalidates only ~1/N of cache entries — not the entire cache — keeping hit rates high during scaling and failures. Google's consistent hashing with bounded loads caps per-node load on top.

Flashcards (Spaced Repetition)

Q: In one sentence, what does consistent hashing do that mod N doesn't?
A: It ties a key's home to a position on a ring instead of the machine count, so adding/removing a machine moves only ~1/N of keys (splitting one arc) instead of reshuffling nearly everything.
Q: How do you find which machine owns a key on the ring?
A: Hash the key to a point on the ring, then take the first machine at or after that point going clockwise (wrapping to the first machine if you pass the end). Implemented as a binary search over sorted machine positions — O(log N).
Q: What problem do virtual nodes solve, and how?
A: They fix uneven load (and lumpy failure). Placing each machine at many points (~100–200) means it owns many small scattered arcs that average to a near-equal share, and when it dies its arcs spread across many neighbors instead of doubling one.
Q: Does consistent hashing solve the hot-key problem?
A: No. It evenly spreads distinct keys, but a single hot key hashes to one point and lands on one machine. Hot keys are still handled app-side — replicating or caching the key (Day 6).
Q: How is consistent hashing combined with replication in Dynamo-style systems?
A: A key is stored on the next R machines clockwise from its point on the ring, not just the first. That gives redundancy (survives node loss) and plugs into quorum reads/writes (Day 10).

If you only remember one thing: bend the keyspace into a ring, put machines and keys on it, and send each key to the next machine clockwise. Now adding or losing a machine disturbs only the one arc beside it — about 1/N of the data — instead of the whole cluster, which is what made elastic, always-on distributed storage possible. Sprinkle in virtual nodes so the arcs come out even and failures spread gently, and you've got the quiet circle running underneath DynamoDB, Cassandra, CDNs, and half the systems you'll design.

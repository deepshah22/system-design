Consistent Hashing
Add a Machine Without Moving the World

Yesterday you cut your data into shards and immediately hit a wall: with plain hash(key) mod N, adding a single machine changes the answer for almost every key, forcing the cluster to shove ~80% of its data across the network while still serving traffic. That's not a growing pain — it's a reason teams are terrified to scale. Consistent hashing is the beautiful idea that fixes it: lay your machines out on a ring so that adding or removing one only disturbs its immediate neighbor. Grow the cluster by 10%, move about 10% of the data — no more, no less.

Visual Memory Anchor

🕛⭕

Picture this

Imagine a giant clock face. Your machines are pinned onto the rim at different hours — one at 12, one at 4, one at 8. Now every piece of data also gets a spot on the same rim (wherever its hash lands). To find which machine owns a piece, you drop it onto the rim and walk clockwise until you bump into the first machine — that machine owns it. Here's the magic: if you add a new machine at, say, 2 o'clock, the only data that changes hands is the slice sitting between 12 and 2 — the new machine quietly steals that arc from its neighbor, and every other machine keeps exactly what it had. No cluster-wide reshuffle. Picture that clock, the clockwise walk, and one new hand sliding in to grab just the wedge behind it — that single image is consistent hashing.

The Story

Database Dave walked away from yesterday with a working sharded cluster and one terrifying number burned into his memory: when he added a 5th machine to his 4-shard setup, roughly 80% of his keys had to move. His hash(key) mod N scheme was balanced and fast — right up until the moment he needed to grow, at which point it turned every capacity upgrade into a white-knuckle, all-night data migration. His teammates started avoiding scaling entirely, over-provisioning huge machines just to never touch N again. That's the tell of a broken design: people bending the whole system to avoid one operation.

A senior engineer drew a circle on the whiteboard. "Stop thinking of shards as slots numbered 0 to N-1," she said. "Think of one big ring of hash values, from zero wrapping all the way back to zero. Hash each machine to a point on the ring. Hash each key to a point on the ring too. A key belongs to the first machine you hit walking clockwise from it." Dave squinted. "So where a key lives no longer depends on how many machines there are — only on where the machines sit on the ring." Exactly. And that one change is everything: adding a machine drops a new point on the ring, and it only takes over the arc between itself and the previous machine. Every key outside that arc doesn't move at all.

Dave rebuilt his router around the ring and added a 6th machine live. This time the monitoring showed only ~17% of keys moving — the one arc the new machine claimed — and the rest of the cluster never noticed. But he hit one snag: with only a handful of machines dropped randomly on the ring, the arcs came out lopsided. One machine landed in a big empty stretch and ended up owning nearly half the data. The senior engineer smiled — "That's the last piece. We don't place each machine once; we place it a hundred times, as a hundred tiny points scattered around the ring. The arcs average out, and load goes flat." Those hundred phantom points are called virtual nodes, and they're the difference between consistent hashing in theory and consistent hashing that actually works.

What should stay in your head
Plain mod N ties a key's home to the count of machines, so changing the count re-homes almost everything. Consistent hashing ties a key's home to a machine's position on a ring instead — so adding or removing a machine only re-homes the keys in one arc, about 1/N of the data, and everything else stays put. The price of admission is that a few randomly-placed machines create lopsided arcs, so you give each machine many virtual nodes (points scattered around the ring) to smooth the load out. Ring + clockwise walk + virtual nodes = elastic, always-on sharding.

How the Ring Actually Works

Step 1 — turn the hash space into a circle

A hash function like MD5 or MurmurHash turns any input into a big number in a fixed range — say 0 to 232−1. Normally we'd take that number mod N to pick a shard. Consistent hashing does something different: it treats the whole range as a circle, where the largest value wraps back around to 0, like the 12 on a clock sitting next to the 1. Nothing physical changes — it's the same hash numbers — we've just agreed to think of them as points on a ring instead of a line.

Step 2 — place the machines on the ring

Each machine gets hashed too. You feed something stable about it — its name or IP, e.g. hash("node-A") — into the same function, and that lands the machine at a point on the ring. Three machines might land at positions equivalent to 12 o'clock, 4 o'clock, and 8 o'clock. These positions are what the whole scheme hangs on, and crucially they don't depend on how many other machines exist — hash("node-A") is the same whether the cluster has 3 machines or 300.

Step 3 — a key belongs to the next machine clockwise

To find a key's home, hash the key to a point on the ring, then walk clockwise until you reach the first machine. That machine owns the key. In code it's a sorted list of machine positions and a binary search for "the first position ≥ the key's position" (wrapping to the first machine if you run off the end). Each machine therefore owns the arc of the ring between the previous machine and itself.

Lookup.hash the key → binary-search the sorted ring for the next node clockwise. O(log V) where V is the number of (virtual) node points. No coordination, no central lookup on the hot path.
Add a machine.drop its point(s) on the ring. It claims the keys in the arc just behind it — those, and only those, migrate from the single neighbor that used to own that arc. Everyone else is untouched.
Remove a machine.delete its point(s). Its arc simply falls to the next machine clockwise. Again, only that one neighbor gains keys; the rest of the ring doesn't move.

Why only 1/N moves

Here's the intuition in one line. With N machines evenly spread, each owns about 1/N of the ring. Add one machine and it carves out one fresh arc of roughly 1/(N+1) of the ring — taken entirely from the single machine whose territory it landed in. So only that ~1/(N+1) slice of keys changes owner; the other N−1 machines don't move a single byte. Compare that to mod N, where bumping N to N+1 changes the remainder for nearly every key at once. Same goal — spread keys over machines — but one design makes growth proportional and the other makes it catastrophic.

Virtual Nodes — The Part That Makes It Actually Work

The plain ring has a quiet flaw you saw in the simulator at "1 virtual node": with only a few points dropped at random, the arcs come out uneven. One machine lands in a wide-open stretch and owns 40% of the ring while another owns 8%. Randomness is lumpy at small counts. Worse, when a machine dies, its entire arc dumps onto the single next machine — instantly doubling that one neighbor's load instead of spreading the pain.

The fix is virtual nodes (also called vnodes or replicas-on-the-ring): instead of placing each machine at one point, place it at many — hash("node-A#1"), hash("node-A#2"), … hash("node-A#150") — all owned by the same physical machine. Now each machine is represented by a spray of points all around the ring, and two good things happen:

Load evens out.Averaging 150 small random arcs per machine gives a far smoother total than one big random arc. Load imbalance shrinks roughly with 1/√(vnodes) — which is exactly why the bars in the sim flatten as you raise the slider.
Failure spreads out.When a machine dies, its 150 little arcs each fall to a different neighbor, so the departed load is shared across the whole cluster instead of crushing one node.
Heterogeneous hardware.Give a machine twice the RAM? Give it twice the vnodes. The number of points is a clean dial for "how much of the ring should this machine own."

The cost is modest: more points means a slightly bigger ring structure and more memory for the map, but lookups are still O(log V). In practice systems use anywhere from a handful to a few hundred vnodes per machine. This is why "consistent hashing" in the wild almost always means "consistent hashing with virtual nodes" — the bare ring is a teaching tool; vnodes are the production version.

Variants & Alternatives (There's More Than One Ring)

Classic ring + vnodes

The Amazon Dynamo design: a hash ring where each node holds many virtual nodes. Simple, elastic, battle-tested. The default mental model and what most systems mean by "consistent hashing."

Great for: general elastic sharding. Used by: Cassandra, DynamoDB, Riak, memcached (ketama).

Rendezvous (HRW) hashing

Skip the ring entirely: for a key, compute hash(key + node) for every node and pick the highest score. Same ~1/N movement guarantee, naturally weighted, and no ring structure to maintain — at the cost of an O(N) scan per lookup (fine for small N).

Great for: smaller node counts, weighted routing. Used by: some CDNs, GlusterFS.

Jump consistent hash

A tiny, allocation-free algorithm (a few lines of math) that maps a key to one of N buckets with perfectly even load and minimal movement — but buckets are numbered 0..N-1, so you can only add/remove at the end, not remove an arbitrary node.

Great for: fixed-identity shards, sharded storage where you only grow. From: Google.

Bounded-load consistent hashing

Classic hashing can still let one node run hot under skewed traffic. This variant adds a cap: if the clockwise-next node is already at its load limit, the key spills to the following node. Guarantees no node exceeds ~(1+ε) of average.

Great for: request routing with hot keys. Used by: Google's load balancers, Vimeo.

The through-line: they all chase the same two properties — even load and minimal movement when the node set changes. Which one you pick depends on how big N is, whether nodes have different capacities, and whether you need to remove arbitrary nodes or only grow.

Common Mistakes & Misconceptions

The misconceptionThe reality
"Consistent hashing makes load perfectly even."The bare ring with one point per node is actually quite lumpy — some nodes can own 40%+ of the ring. Even load comes from virtual nodes, not the ring itself. Without them the scheme is barely usable.
"It's called consistent hashing, so it gives strong consistency (CAP)."Different word. Here "consistent" means the key→node mapping stays stable as nodes change. It has nothing to do with the C in CAP (Day 10) — you still choose your consistency model separately.
"The ring solves the hot-key problem too."No. A single hot key still hashes to exactly one arc, so it still lands on one node. Consistent hashing balances many keys and makes scaling cheap — hot keys still need caching or key-splitting (Day 13), or bounded-load variants.
"Adding a node moves exactly 1/N of the data."It's ~1/N on average. Which keys move depends on where the new node's points land; with few vnodes the actual amount is noisy. More vnodes make it both smaller-variance and closer to the ideal.
"More virtual nodes is always strictly better."Load smooths out, but the ring structure grows and lookups/metadata cost a bit more. There's a sweet spot (often tens to a few hundred), not "infinity."
"You can just swap in consistent hashing and be done."Migrating a live cluster from mod N to a ring still means moving data once, and you must keep the ring/vnode map consistent across all routers as nodes join and leave — that coordination is real work.

Active Recall

❓ You run a distributed cache with 10 nodes using hash(key) mod 10. One node dies. What happens to your cache hit rate, why, and how would a hash ring with virtual nodes change the outcome? (click to reveal)
With mod 10, losing a node drops you to mod 9, which changes the target node for ~90% of keys. Almost every lookup now hits a node that never cached that key, so your hit rate collapses to near zero and the whole fleet stampedes the database at once (a cache-avalanche / thundering herd). With a hash ring, the dead node's keys fall only to its clockwise neighbors, so just ~1/10 of keys need re-fetching — the other 90% still hit warm caches. Add virtual nodes and that dead node's load spreads across all surviving nodes evenly instead of dumping onto one, so no single survivor gets crushed. This exact scenario — resilient cache membership — is one of the original motivations for consistent hashing (Akamai's CDN, 1997).

Real-World Engineering

🪶 Amazon DynamoDB / Cassandra

Both descend from the 2007 Dynamo paper, which put consistent hashing at its core: nodes sit on a ring with virtual nodes ("tokens"), each key's arc is replicated to the next few nodes clockwise, and nodes join/leave with minimal data movement. This is the canonical production use.

🌐 Akamai CDN

Consistent hashing was invented here (MIT, 1997) to map web objects to caching servers so that adding or losing an edge server didn't blow away the entire cache. Every CDN since routes requests to cache nodes this way.

💬 Discord

Discord routes millions of live voice/gateway sessions and uses consistent hashing (with rendezvous/HRW hashing for some layers) to map users to service nodes so that scaling the fleet doesn't reshuffle everyone's live connection.

🗃️ memcached clients (ketama)

The classic ketama algorithm made consistent hashing the default for sharded memcached: add or remove a cache server and only a small slice of keys remap, so the cache stays mostly warm instead of cold-flushing on every membership change.

Flashcards (Spaced Repetition)

Q: In one sentence, what problem does consistent hashing solve?
A: It lets you add or remove machines from a sharded cluster while moving only ~1/N of the data (one arc), instead of the ~(N-1)/N cluster-wide reshuffle that hash(key) mod N forces on every resize.
Q: How does a lookup work on the ring?
A: Hash the key to a point on the ring, then walk clockwise to the first node point you reach — that node owns the key. Implemented as a binary search over the sorted list of node positions (wrapping to the first if you run past the end).
Q: What are virtual nodes and why are they essential?
A: Each physical machine is placed on the ring at many points (e.g. 150) instead of one. This averages out the random arc sizes so load is even, and when a node dies its many small arcs spread across all neighbors instead of dumping onto one. The bare one-point ring is too lumpy to use in production.
Q: Does consistent hashing fix the hot-key problem?
A: No. A single hot key still hashes to one arc and lands on one node. Consistent hashing balances many keys and makes scaling cheap; hot keys still need caching, key-splitting, or a bounded-load variant.
Q: Why is "consistent" in consistent hashing NOT the C in CAP?
A: Here "consistent" means the key→node mapping stays stable as the node set changes (minimal remapping). It's unrelated to CAP consistency (whether reads see the latest write) — you still pick your consistency model separately.

If you only remember one thing: mod N ties a key's home to how many machines exist, so growing the cluster re-homes almost everything; consistent hashing ties it to where machines sit on a ring, so growing only re-homes one arc — about 1/N. Add virtual nodes to make the arcs even and failures spread out, and you have the elastic, always-on sharding that every Dynamo-style store, CDN, and distributed cache is built on.

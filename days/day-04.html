Content Delivery Networks (CDN)

CDNs are one of the highest-leverage tools in a system designer's toolkit — they reduce latency by moving content close to users, absorb traffic spikes, terminate TLS at the edge, and protect origins from DDoS. Today you'll master CDN architecture deeply enough to reason about it in any interview.

Why CDNs Exist: The Physics Problem

Every distributed system eventually hits a wall that no amount of code can fix: the speed of light. Light travels at approximately 200,000 km/s through fiber optic cable (about two-thirds the speed in vacuum). That hard physical constant determines the minimum round-trip time between any two points on Earth.

Consider these raw physics latencies:

Route Distance Min RTT (physics) Actual RTT
New York → London 5,600 km 56 ms ~70 ms
New York → Tokyo 10,800 km 108 ms ~150 ms
New York → Sydney 16,200 km 162 ms ~200 ms
New York → São Paulo 7,600 km 76 ms ~110 ms
New York → New York (CDN edge) ~30 km 0.3 ms ~2 ms

If your origin server is in AWS us-east-1 (Virginia) and a user is in Tokyo, the TCP handshake alone costs 150ms — before a single byte of your response is sent. Add TLS (1 RTT for TLS 1.3) and you're at 300ms just for connection setup.

A CDN solves this by placing servers at ~200+ locations worldwide. The Tokyo user connects to a CDN node in Tokyo (~2ms away), which either serves cached content instantly or fetches from origin over a pre-warmed, persistent connection. The physics problem is reduced from 150ms to 2ms for the user-facing part of the journey.

💡 The Interview Framing

When an interviewer asks "how would you reduce latency for global users?" — CDN is almost always part of the answer. But the sophisticated answer explains why: moving content geographically closer to users is the only way to beat physics. No amount of application optimization helps a user in Sydney talking to a server in Virginia.

CDN Architecture

A CDN is a two-tier (or three-tier) hierarchy of servers designed to cache content at the edge.

Excalidraw — CDN Global Architecture

The Three Tiers

1. Edge Nodes (PoPs)

Points of Presence (PoPs) are the CDN's frontline servers, deployed in 100–400+ cities globally. These are the servers your users actually talk to. They:

  • Terminate TCP and TLS connections from end users
  • Serve content directly from cache on a hit
  • Forward requests upstream on a miss
  • Return responses to users and populate the local cache

2. Origin Shield (Mid-Tier / Shield Layer)

A second tier of larger caches, usually in 10–30 regional hub locations. The origin shield is optional but highly recommended for high-traffic origins. When multiple edge nodes in a region all miss cache at the same time (a thundering herd problem), they don't all rush to origin simultaneously — they all forward to the regional shield, which makes a single request upstream. This dramatically reduces origin load.

3. Origin Servers

Your actual application servers, typically in 1–3 regions. The CDN protects them from direct user traffic — only cache misses and non-cacheable requests reach origin. With a well-tuned CDN, >90% of traffic is served from edge, meaning your origin only handles 10% of total requests.

⚠️ Origin Shield is Underused

Most engineers know about edge caching but skip the shield layer. In an interview, mentioning origin shield (and the thundering herd problem it solves) signals you understand CDNs at a production depth, not just conceptually. CloudFront calls it "Origin Shield," Fastly calls it "shielding," Cloudflare calls it "Argo Tiered Caching."

The CDN Request Lifecycle

Excalidraw — CDN Cache Hit vs Cache Miss

Cache HIT Path (fast)

  1. User's browser sends request: GET cdn.example.com/logo.png
  2. DNS resolves to nearest CDN PoP IP (2–10ms)
  3. TCP+TLS connects to PoP (2–5ms, same city)
  4. PoP checks its cache → HIT
  5. PoP returns cached response immediately
  6. Total: 10–30ms (user barely notices)

Cache MISS Path (slower)

  1. Same steps 1–3 above
  2. PoP checks its cache → MISS
  3. PoP forwards request to origin shield (if configured)
  4. Shield checks its cache → MISS, forwards to origin
  5. Origin processes request and returns response (50–200ms)
  6. Shield caches the response, forwards to PoP
  7. PoP caches the response, returns to user
  8. Total: 150–400ms (first user in region pays this cost)

The ratio of hits to total requests is the cache hit ratio. A typical production CDN targets >85% hit ratio for static assets. If your hit ratio is below 70%, you're likely:

  • Over-personalizing responses (per-user cache keys)
  • Setting TTLs too low
  • Using too many query parameters that bust cache
  • Serving content that genuinely changes per-user (authenticated responses)

Push CDN vs Pull CDN

There are two models for how CDN caches get populated. The choice matters for your architecture.

⬇️ Pull CDN
Content fetched on first miss
  • CDN fetches from origin when first requested
  • No action needed on your side
  • First user in each region gets a cache miss
  • Cache fills lazily over time
  • Works well for unpredictable access patterns
  • Used by: CloudFront, Cloudflare (default)
⬆️ Push CDN
You pre-upload content to CDN
  • You push content to CDN edge nodes proactively
  • First user always gets a cache hit
  • Requires integration with deploy pipeline
  • Must manually manage content lifecycle
  • Good for large, predictable files (video, installers)
  • Used by: AWS S3 + CloudFront pre-warming, MaxCDN
💡 The Hybrid Approach (What Production Systems Use)

Most production systems use pull CDN by default, but add a cache pre-warming step to their deployment pipeline for business-critical assets. After deploying a new version of the homepage, a deploy script calls curl from 10 geo-distributed machines to pre-warm caches in all major regions before announcing the deploy. This gives pull-CDN simplicity with push-CDN cold-start protection.

Cache-Control Headers — Know These Cold

Cache behavior is controlled entirely by HTTP headers. Misunderstanding these is the root cause of most CDN bugs in production. Here are the ones every system designer must know:

The Core Headers

Cache-Control: max-age=3600
Cache for 3600 seconds (1 hour). Both browser AND CDN honor this.
Cache-Control: public
Any intermediate cache (CDN, proxy, browser) is allowed to cache this response. Required for CDN caching.
Cache-Control: private
Only the end user's browser can cache. CDN must NOT cache. Use for authenticated/personalized responses.
Cache-Control: no-cache
Can cache, but must revalidate with origin before serving. CDN sends conditional request with ETag/Last-Modified. If unchanged (304), serves cached copy.
Cache-Control: no-store
Never cache, never store. Used for sensitive data (bank statements, medical records).
Cache-Control: s-maxage=86400
Override max-age for shared caches (CDN) only. Browser still uses max-age. Lets you set CDN TTL=1day but browser TTL=5min.
Cache-Control: stale-while-revalidate=60
Serve stale content for up to 60s while refreshing in background. Eliminates cache-miss latency spikes at TTL expiry.
Surrogate-Control: max-age=86400
CDN-specific header (stripped before sending to browser). Lets you set longer CDN TTL independently of browser TTL.
Vary: Accept-Encoding
Create separate cache entries per value of the Accept-Encoding header. Used to cache gzip and identity versions separately. Warning: Vary: User-Agent creates millions of cache keys — avoid it.

Practical TTL Strategy

HTML pages
0–60s (or no-cache)
API responses
0s–5min
CSS / JS (versioned)
1 year (31536000s)
Images (versioned)
6–12 months
Fonts
1 year
Video segments
1 year (content-addressed)

The pattern for CSS/JS/images is: hash the file content into the filename (e.g., main.a3f2b1c9.js), then set Cache-Control: public, max-age=31536000, immutable. When the file changes, the hash changes, creating a new URL — no cache invalidation needed.

Cache Invalidation: The Hard Part

There are two hard problems in computer science: cache invalidation, naming things, and off-by-one errors. Cache invalidation is genuinely hard because CDN edge nodes are distributed — clearing a cache entry means propagating that operation to 200+ locations globally.

Strategy 1: URL-Based Versioning (Best)

Embed a version or content hash in the URL. When content changes, the URL changes, so old cached content is never "invalidated" — it's simply abandoned at the old URL. 

# Before deploy
https://cdn.example.com/app.js?v=1.2.0
Cache-Control: public, max-age=86400

# After deploy (new content hash → new URL)
https://cdn.example.com/app.a3f2b1c9.js
Cache-Control: public, max-age=31536000, immutable

# Old URL can still be cached indefinitely — no one requests it

This is used by virtually every large-scale frontend build system (Webpack, Vite, Parcel). The HTML page that references these URLs has a short TTL so it picks up the new asset URLs quickly.

Strategy 2: TTL Expiry (Simple)

Set a short enough TTL that stale content expires naturally. The trade-off: short TTL = more origin traffic, more cache misses, more latency. This is the lazy approach — it works but wastes CDN capacity.

Strategy 3: Active Cache Purge (Targeted)

Use the CDN's API to immediately invalidate specific URL patterns globally. Every major CDN supports this: 

# CloudFront API call — invalidate all JPG files
aws cloudfront create-invalidation \
  --distribution-id E2QWRUHEXAMPLE \
  --paths "/*.jpg"

# Cloudflare API — purge specific URLs
curl -X POST "https://api.cloudflare.com/client/v4/zones/{zone_id}/purge_cache" \
  -H "Authorization: Bearer {token}" \
  -d '{"files":["https://example.com/image.jpg"]}'
⚠️ Purge Propagation Delay

Even "instant" invalidation takes time to propagate to all edge nodes. CloudFront guarantees files are removed in 10–30 seconds. Cloudflare is typically faster (near-instant via Raft consensus across PoPs). During this window, some users may still see stale content. Design for this by using URL versioning for assets that cannot tolerate staleness.

Cache Key Design

A cache key determines when two requests are considered "the same" and can share a cached response. By default, most CDNs key on URL path + query string. You can customize this:

  • Exclude tracking params: Strip utm_source, fbclid from the key to avoid blowing up your cache
  • Include headers: Cache separate versions by Accept-Language or Accept-Encoding
  • Normalize: Ensure /page?a=1&b=2 and /page?b=2&a=1 map to the same key

Anycast & Geo-Routing: How Users Find the Nearest PoP

A CDN with 300 PoPs is useless if users keep getting routed to the wrong one. Two routing techniques ensure users always reach the nearest edge node:

Anycast Routing (BGP-based)

The CDN advertises the same IP address from multiple physical locations via BGP. Internet routers, following BGP's shortest-path algorithm, naturally route packets to the closest PoP announcing that IP. When a PoP goes offline, BGP withdraws its announcement and traffic automatically reroutes to the next closest PoP — no DNS change, no configuration, no TTL wait.

Cloudflare uses anycast for all its PoPs. Every Cloudflare PoP announces the same IP space. This is why Cloudflare failover is near-instantaneous — BGP convergence takes seconds, not minutes.

DNS-Based Geo-Routing

The CDN's DNS server detects the client's geographic location (via their resolver's source IP) and returns a different A record pointing to the nearest PoP. Unlike anycast, failover requires a DNS change + TTL wait, but it offers more granular control.

AWS CloudFront and Akamai primarily use DNS geo-routing. Route 53 health checks combined with latency-based routing give granular control over which region users are sent to.

Property Anycast DNS Geo-Routing
Failover speed Seconds (BGP convergence) Minutes (DNS TTL expiry)
Routing control Implicit (BGP path costs) Explicit (per-region rules)
DDoS resilience Excellent (traffic spreads) Good
Complexity Requires own ASN + BGP Simple DNS config
Used by Cloudflare, Google CloudFront, Akamai

CDN for Dynamic Content — Beyond Static Files

A common misconception is that CDNs only help for static files (images, CSS, JS). In reality, CDNs accelerate dynamic content in three ways:

1. TLS Termination at Edge

Without a CDN, a user in Tokyo making HTTPS requests to a server in Virginia pays the TLS handshake cost of ~300ms (1 RTT to Virginia and back). With a CDN, TLS terminates at the Tokyo PoP (~2ms away), and the CDN maintains a persistent, already-handshaked connection to the origin. The user sees ~4ms for TLS instead of 300ms — even for fully dynamic, uncacheable requests.

2. TCP Connection Pooling to Origin

CDN edge nodes maintain a pool of warm TCP connections to origin servers. When a user request misses cache, the edge reuses an existing connection (no new TCP 3-way handshake to origin), reducing miss latency by another 50–150ms.

3. Edge Compute (Serverless at Edge)

Modern CDNs let you run JavaScript/WASM at the edge node, eliminating the origin roundtrip entirely for logic that doesn't need your full backend:

  • Cloudflare Workers: V8 isolates at ~300 edge locations, sub-millisecond cold starts
  • CloudFront Functions / Lambda@Edge: JWT verification, request rewriting, A/B testing at edge
  • Fastly Compute@Edge: WebAssembly at edge, supports Rust, Go, JavaScript
// Cloudflare Worker: Validate JWT at edge before hitting origin
export default {
  async fetch(request, env) {
    const token = request.headers.get('Authorization')?.split(' ')[1];
    if (!token || !isValidJWT(token, env.JWT_SECRET)) {
      return new Response('Unauthorized', { status: 401 });
    }
    // Valid — forward to origin with the 150ms latency saved for invalid requests
    return fetch(request);
  }
}
✅ Interview Gold: API Acceleration

When asked "how do you reduce API latency globally?" — a senior answer is: (1) Cache API responses at CDN for endpoints that allow it (public data, shared responses), (2) Use CDN for TLS termination and TCP pooling even for dynamic requests, (3) Use edge compute for auth, routing, and personalization logic that doesn't need the full origin. This can cut global P99 latency from 500ms to 50ms for many API patterns.

💡 Interview Tip: You Don't Need to Pick One

In interviews, defaulting to "I'd use Cloudflare for static assets and TLS termination, with CloudFront for media files close to S3" is a perfectly valid answer. More important than knowing specific provider features is understanding the concept: edge caching, cache hierarchy, TTL strategy, and when to CDN vs. not CDN.

When to Use a CDN — and When Not To

✅ Use CDN For
High ROI scenarios
  • Static assets (JS, CSS, images, fonts)
  • Video/audio files and adaptive bitrate streaming
  • Software downloads (installers, packages)
  • Public APIs with shared responses
  • Any content served to a global user base
  • DDoS mitigation layer in front of origin
  • TLS termination for all traffic (even dynamic)
  • High-traffic sites that need to shield origin
❌ Don't CDN For
Low ROI or problematic scenarios
  • Highly personalized responses (per-user data)
  • Financial transactions requiring real-time consistency
  • Internal-only services (no public internet users)
  • Real-time data streams (use WebSockets directly)
  • User-uploaded content served only to uploader
  • Machine-to-machine APIs in same datacenter
⚠️ Authentication and CDNs

Never serve authenticated/personalized content through a CDN without careful configuration. A misconfigured CDN can serve User A's private data to User B if the cache key doesn't include the user identifier. Always set Cache-Control: private for responses that contain personal data, or use signed URLs / cookies for CDN-served authenticated content.

System Design Interview Impact

CDNs appear in almost every major system design interview question. Here's how to weave CDN into common interview designs:

Design YouTube / Netflix

Video content is served entirely via CDN. The key insight: Netflix uses its own CDN (Open Connect) embedded directly in ISP networks. Instead of a user request leaving the ISP to reach Netflix, the ISP itself caches the most popular Netflix content and serves it locally. This reduces Netflix's egress bandwidth costs and gives users the best possible performance.

Design Twitter / Instagram (Feed)

All media (photos, videos, thumbnails) served via CDN. The home feed HTML is not CDN-cacheable (personalized), but images referenced by the feed are cached globally. Profile pictures use a long TTL with URL versioning — when a user updates their avatar, the URL changes (new hash), and the old URL expires naturally.

Design a Global E-Commerce Platform

Product images, catalog pages, and marketing pages via CDN with 5–60 minute TTL. The shopping cart and checkout are NOT cached (transactional). Pricing and inventory APIs have short TTL (30 seconds) to balance freshness with origin protection during sale events where traffic spikes 100x in seconds.

Back-of-Envelope: CDN Impact Numbers

In your interview estimates, use these rules of thumb:

  • CDN reduces origin bandwidth by 70–95% for typical static-heavy sites
  • CDN reduces global P50 latency from ~200ms to ~20ms for cached content
  • CDN improves global P99 latency from ~2000ms to ~200ms
  • Cache hit ratio goal: >85% for static, >30% for semi-dynamic
  • CloudFront pricing: ~$0.0085/GB out (US), $0.01/10,000 HTTPS requests

✅ Key Takeaways for Day 4

  • Physics limits latency — a server in Virginia can never respond in under 100ms for a user in Tokyo. CDNs beat this by placing caches in 200+ locations globally.
  • Three-tier hierarchy — Edge nodes (PoPs) serve users; origin shield aggregates misses from regional edges; origin servers handle only what can't be cached.
  • Pull CDN is the default — cache fills on first miss. Push CDN is for known, large files. Hybrid (pull + pre-warming) is what production systems do.
  • Cache-Control is everything — public + s-maxage for CDN, private for authenticated data, immutable for versioned assets. Know these without thinking.
  • URL versioning beats active invalidation — embed content hash in filenames (app.a3f2b1c9.js), set TTL=1 year, never think about invalidation again.
  • CDNs help dynamic content too — TLS termination at edge saves 150ms+, connection pooling eliminates TCP setup overhead, edge compute moves logic to the edge.
  • Cloudflare anycast vs. CloudFront DNS-routing — anycast routes via BGP (instant failover), DNS-routing is explicit but TTL-delayed. Both work; context determines which to use.
  • In interviews — mention CDN as a first-class component for any public-facing system. Mentioning origin shield and cache hit ratio optimization signals production depth.

🧠 Practice Questions

1
Your e-commerce site serves product images that rarely change (maybe once per season). You also serve real-time inventory counts (updates every second). Design the CDN strategy for both. What TTLs and Cache-Control headers do you set?
Hint: Images get URL versioning + immutable TTL. Inventory may bypass CDN or use stale-while-revalidate with very short TTL.
2
Your CDN cache hit ratio is only 45% even though 90% of your traffic is for static assets. List 5 things you'd investigate to find the root cause.
Hint: Cache key pollution (query params), TTL too low, Vary header issues, too many unique URLs, authentication headers bypassing cache.
3
A junior engineer suggests: "Let's set Cache-Control: public, max-age=3600 on all API responses including the /user/profile endpoint." What specific security incident could this cause, and how would you fix it?
Hint: CDN caches User A's profile response. Next user requesting /user/profile gets User A's data from cache. Fix: require private or add user ID to cache key.
4
You're designing Netflix's video delivery infrastructure. Netflix needs to serve 4K video (~25 Mbps bitrate) to 230 million subscribers simultaneously, at peak with ~20 million concurrent streams. How would you architect this? Why did Netflix build its own CDN (Open Connect) instead of using Cloudflare or CloudFront?
Hint: 20M streams × 25 Mbps = 500 Tbps peak. Egress costs at that scale are prohibitive with public CDNs. Netflix negotiates free ISP colocation in exchange for reducing ISP's upstream bandwidth costs.
5
Explain how Cloudflare Workers could be used to implement A/B testing at the edge, without any origin server involvement. What are the limitations of this approach?
Hint: Workers can read/set cookies, modify responses, and forward to different origin paths. Limitations: no database access, no session state, limited execution time.

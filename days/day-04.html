Content Delivery Networks
How the Internet Goes Fast

A CDN is one of the first scaling solutions any large system deploys. Understanding how edge caching works — and crucially, how it fails — is essential knowledge for every system design interview.

1. Why CDNs Exist

The speed of light is ~200,000 km/s through fiber (refractive index ≈ 1.5). A round-trip from New York to London is roughly 5,600 km of cable — that's ~28 ms of pure propagation delay, before any server processing. A user in Mumbai querying a server in Virginia adds ~140 ms per request.

CDNs solve three distinct problems:

  • Latency — serve bytes from a server physically close to the user
  • Throughput — distribute traffic across hundreds of edge servers, reducing origin load
  • Availability — edge nodes absorb traffic spikes; if origin goes down, stale cache keeps serving
Interview framing
When an interviewer asks "how do you serve a billion users?", CDN is almost always your first tool. Justify it with latency numbers — examiners want to see you reason about physics, not just name-drop Cloudflare.

On a miss, the edge fetches from origin, stores the response, and serves the user — subsequent requests from that PoP are served from cache.

Anycast DNS — How the Nearest Edge is Selected

CDN providers announce the same IP block from every PoP using BGP Anycast. When you query img.example.com, the CDN's authoritative DNS returns an IP that your ISP routes to the topographically nearest PoP automatically — no geolocation database lookup needed.

Why Anycast matters
Anycast is also how DDoS mitigation works at scale. An attack targeting one IP is spread across every PoP in the anycast group, diluting the traffic. Cloudflare absorbs terabit-scale attacks this way.

3. CDN Architecture — PoPs and Tiers

Points of Presence (PoPs)

A PoP is a physical data center the CDN operator owns or leases, typically colocated in large IXPs (Internet Exchange Points) like Equinix. Major CDNs have 200–300+ PoPs globally. Each PoP contains:

  • Edge servers — HTTP servers with large NVMe SSD caches
  • Reverse proxy — terminates TLS, evaluates cache rules
  • Load balancer — spreads requests across edge servers in the PoP

Tiered Caching (Shield / Mid-Tier)

Large CDNs add a "shield" tier — a regional cluster sitting between edge PoPs and origin. If 50 PoPs all miss on the same URL simultaneously, without a shield each sends a separate request to origin (50 requests). With a shield, all 50 forward to one shield node, which fetches once from origin.

4. Cache-Control Headers

The CDN reads HTTP response headers from your origin to decide how (and whether) to cache. Getting these right is as important as any CDN architecture decision.

Key Directives

# Cache publicly for 1 year, allow stale for 1 hour if origin is unreachable
Cache-Control: public, max-age=31536000, stale-while-revalidate=3600

# Private — never cache (user-specific data, auth responses)
Cache-Control: private, no-store

# Cache but revalidate before serving (ETag / Last-Modified check)
Cache-Control: public, no-cache, max-age=0

# CDN-specific: cache at edge for 1 day, but tell browser only 1 hour
Cache-Control: public, s-maxage=86400, max-age=3600

# Surrogate headers (Varnish / Fastly)
Surrogate-Control: max-age=86400
DirectiveWho It AffectsMeaning
publicCDN + browserResponse may be cached by shared caches
privateBrowser onlyOnly the end user's browser may cache it
max-age=NBrowserTreat as fresh for N seconds
s-maxage=NCDN onlyOverrides max-age for shared caches
no-cacheBothMust revalidate with origin before use
no-storeBothNever cache at all (auth tokens, PII)
stale-while-revalidateBothServe stale while fetching fresh in background
stale-if-errorBothServe stale if origin returns 5xx
Common mistake
Setting Cache-Control: no-cache does NOT mean "don't cache." It means "always revalidate before serving." Use no-store if you genuinely want nothing cached. Confusing these two costs precious interview points.

Cache Vary

The Vary header tells the CDN to cache separate versions based on request headers. Vary: Accept-Encoding caches separate gzip and br versions. Vary: Accept-Language caches per locale.

Avoid Vary: Cookie or Vary: Authorization
These fragment your cache into millions of unique entries, destroying hit ratios. Instead, separate personalized content into API calls and cache only the static shell.

5. Pull CDN vs Push CDN

Pull CDN (Lazy / On-Demand)

Origin is the source. CDN fetches and caches content on the first request (cache miss). All major CDNs default to this.

Simple setup No pre-warming needed First user after miss is slow Cold cache on deploy

Push CDN (Pre-Population)

You proactively upload assets to edge nodes before any user requests them. Common for large media files (video, software downloads).

Zero cold-start misses Predictable latency Complex pipeline Costs storage at rest

Write-Through (Dynamic)

Application writes to CDN and origin simultaneously. Used for semi-dynamic content (sports scores, stock quotes with 30s TTL).

Consistent Higher write complexity Stale risk on failure

Stale-While-Revalidate

Serve cached content immediately, refresh in background. Best of both worlds for content that changes occasionally (blogs, product pages).

Fast always No thundering herd Briefly stale
90% of interview cases
Use pull CDN + stale-while-revalidate for static assets, and push CDN for large media files (video segments, software installers). State this explicitly and interviewers will be impressed.

6. Cache Invalidation — The Hard Problem

Phil Karlton famously said: "There are only two hard things in Computer Science: cache invalidation and naming things." CDN invalidation is where systems get messy in interviews.

Strategy 1: TTL-Based Expiry

Let cache entries expire via max-age. Simple, predictable, no API calls needed. Problem: stale content lives for up to TTL after a change.

  • Best for: frequently changing content where some staleness is acceptable
  • Risk: bugs deployed to CDN can't be recalled quickly if TTL is long

Strategy 2: URL Versioning (Cache Busting)

Embed a hash or version in the URL: /assets/app.a3f2b9.js. Old URL still cached forever (TTL = 1 year). New deploy = new URL = new cache entry. 

<!-- Bad: filename never changes, stale for TTL duration -->
<script src="/assets/app.js"></script>

<!-- Good: hash changes with content, old URL safely cached forever -->
<script src="/assets/app.a3f2b9c1.js"></script>
Best practice
Use URL versioning for all static assets. Set max-age=31536000 (1 year) on versioned URLs and let your build pipeline update them. This is how webpack, Vite, and Next.js work by default.

Strategy 3: Purge / Instant Invalidation

Most CDNs expose a purge API (Cloudflare Purge, CloudFront Invalidation, Fastly Instant Purge). You call it after a deploy to immediately evict stale entries.

Surrogate Keys / Cache Tags (Advanced)

Fastly and Cloudflare Enterprise support tagging cached responses with arbitrary keys. You purge by tag rather than URL — one API call clears all pages that contain a specific product, user, or article. 

# Origin sets tag header
Surrogate-Key: product-42 category-electronics homepage

# Later, purge all pages containing product 42
curl -X POST https://api.fastly.com/service/SVC_ID/purge/product-42
Interview application
If asked "how do you update a cached product page when inventory changes?", answer: surrogate keys + purge by tag. This is how large e-commerce sites (Shopify, Wayfair) maintain CDN caching with dynamic data.

Edge-Side Rendering (Modern CDN Compute)

Modern CDNs (Cloudflare Workers, Fastly Compute, Lambda@Edge) let you run JavaScript at the edge. This enables:

  • A/B testing at the edge — no origin round-trip to decide variant
  • Edge authentication — validate JWT before request reaches origin
  • Personalized caching — vary cache by user segment, not individual cookie
  • Partial page caching — assemble page from cached fragments at edge

9. CDN in System Design Interviews

When to Introduce a CDN

Mention a CDN immediately when the problem involves any of these:

  • Serving users in multiple geographic regions
  • Static assets (JS, CSS, images)
  • Video streaming (YouTube, Netflix)
  • Scale > 1M DAU (origin bandwidth cost alone justifies it)
  • High availability requirements (CDN can serve stale during origin downtime)

How to Structure a CDN Section in Your Answer

  1. State the goal: "To reduce latency for global users and offload origin bandwidth"
  2. Choose pull vs push based on content type
  3. Define Cache-Control policy for each content category
  4. Define invalidation strategy for mutable content
  5. Address failure mode: what happens if CDN goes down?

YouTube / Video Platform Example

YouTube video segments are immutable blobs (.mp4 chunks). After upload, they never change. This is a perfect push CDN case:

  • Transcoding pipeline outputs segments to object storage (GCS/S3)
  • CDN pre-warms popular videos from regional distribution servers
  • Segments have Cache-Control: public, max-age=31536000, immutable
  • Cache hit ratio approaches 99% for popular content
  • Origin bandwidth: only new/unpopular videos; CDN absorbs the rest

Twitter / Social Feed Example

Tweets are dynamic (likes, retweets update in real-time), but profile images and media attachments are static. Split the concern:

  • Tweet text/metadata: never cache (or short TTL + surrogate keys)
  • Profile photos, banners: CDN with max-age=3600, URL versioning on upload
  • Embedded images: CDN with immutable URLs (hash in path)
  • JavaScript bundles: max-age=31536000 with content hash

10. CDN Failure Modes

FailureImpactMitigation
CDN PoP outage Users in affected region get origin latency Multi-CDN routing; origin must handle full load
Thundering herd on miss New content: hundreds of PoPs all miss simultaneously, stampeding origin Request coalescing (only one upstream per edge); shield tier
Cache poisoning Attacker injects malicious content via request smuggling Strict header validation; cache only GET/HEAD; no Vary: Cookie
Stale content after deploy Users see old JS/CSS, breaking the app URL versioning (hash) + purge API on deploy
CDN provider outage Cloudflare/Fastly outages have taken down large portions of internet Multi-CDN (Cloudflare + CloudFront); DNS failover to origin
Origin overload during CDN miss storm Warm cache + deploy = millions of simultaneous misses Staggered deploys; blue/green with pre-warming; CDN cache warm scripts
The Fastly Outage (June 2021)
A misconfiguration in Fastly's CDN software took down Reddit, Twitch, GitHub, CNN, The Guardian, and hundreds of others simultaneously for ~1 hour. The incident demonstrated why multi-CDN is worth the complexity for tier-1 services.

Multi-CDN Architecture

Large players (Netflix, Apple, Shopify) route traffic across multiple CDN providers simultaneously using DNS-based load balancing and real-time performance monitoring (Real User Monitoring — RUM). If Fastly P99 latency spikes, traffic shifts to CloudFront.

Practice Questions — CDN
  1. Netflix cache warm: Netflix releases a new season of a show at midnight. Hundreds of millions of users will hit play in the first 10 minutes. How do you ensure the CDN can serve this without overwhelming your origin? Walk through the content pipeline from upload to first user request.
  2. Cache invalidation: Your e-commerce site caches product pages with a 24-hour TTL. A product goes out of stock. Users are still seeing "Add to Cart" for 24 hours. How would you redesign the caching strategy to handle real-time inventory changes without sacrificing cache performance?
  3. Personalization dilemma: Your news homepage is different for each user (personalized recommendations). You want to cache it at the CDN. How would you architect this so most of the page is cached while the personalized section is still dynamic?
  4. Multi-CDN: Your system uses only Cloudflare. An interviewer asks how you'd handle a Cloudflare outage (which happens ~1–2 times/year). Design a multi-CDN failover system. What are the tradeoffs?
  5. Security: Your CDN serves a banking application's static assets. A penetration tester finds that injecting a specific Host header causes the CDN to cache a different user's account page under a shared key. What is this attack called, how did it happen, and how do you fix it?

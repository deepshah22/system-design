Why System Designers Must Understand Networking

At first glance, networking seems like ops or infrastructure territory. But every architectural decision you make as a system designer — choosing between REST and gRPC, deciding whether to use long-polling or WebSockets, deciding where to place a CDN — is a networking decision in disguise.

Here's what networking knowledge buys you in an interview:

  • Latency intuition: Know that a round trip from US East to US West is ~60ms, and that a TLS handshake adds 1–2 round trips — so connection setup alone is 60–120ms
  • Protocol selection: Know why video streaming uses UDP, why file transfer uses TCP, and why modern apps use HTTP/2
  • Failure modes: Know what happens when packets are dropped, when DNS entries are stale, or when a TCP connection silently dies
  • Optimization opportunities: Know that connection pooling exists because TCP setup has overhead, that HTTP/2 multiplexing eliminates the cost of multiple connections, that DNS TTL controls cache staleness
💡 Interview Tip

When asked "how would you reduce latency?" — the best answers always start at the network layer. Moving computation closer to users (CDN, edge), reducing round trips (HTTP/2), keeping connections alive (connection pooling) — these are all network-layer wins that outperform any amount of application optimization.

The TCP/IP Model (4 Layers)

The internet is built on a layered model. Each layer has a specific job and communicates only with the layers directly above and below it. This abstraction is what makes the internet work across wildly different hardware and software.

Excalidraw — TCP/IP Stack with Protocols
4
Application Layer 4
HTTP, HTTPS, DNS, FTP, SMTP, WebSocket, gRPC — protocols that applications speak directly. Data unit: message
3
Transport Layer 3
TCP, UDP — handles end-to-end delivery, ports, reliability and flow control. Data unit: segment (TCP) / datagram (UDP)
2
Internet Layer 2
IP (IPv4/IPv6), ICMP — routes packets between networks using IP addresses. Data unit: packet
1
Network Access Layer 1
Ethernet, Wi-Fi, fiber — physical transmission of bits on a medium. Data unit: frame

The key insight is that each layer adds a header as data travels down the stack (encapsulation), and strips that header as data travels up on the receiving end (decapsulation). A web server's HTTP response is wrapped in a TCP segment, which is wrapped in an IP packet, which is wrapped in an Ethernet frame. The router only cares about the IP layer. The NIC only cares about the frame. The application only cares about the HTTP message.

💡 Why Layer Thinking Matters for System Design

Load balancers operate at different layers: Layer 4 (transport) LBs route by TCP/UDP without looking at content — extremely fast. Layer 7 (application) LBs inspect HTTP headers and can route by path, cookie, or header value — more flexible but more expensive. When you say "I'd put an L7 load balancer in front of the API tier," you're making a networking-informed decision.

TCP vs UDP: The Core Transport Choice

Nearly everything you build runs on either TCP or UDP. Understanding the tradeoffs is not optional — it determines the protocol you choose for every internal and external communication.

TCP: The Reliable Protocol

TCP (Transmission Control Protocol) guarantees that every byte sent is received, in order, exactly once. It achieves this through:

  • 3-way handshake: SYN → SYN-ACK → ACK before any data flows. ~1 RTT overhead.
  • Acknowledgments (ACKs): Receiver confirms each segment; sender retransmits on timeout
  • Ordered delivery: Sequence numbers ensure bytes arrive in the right order
  • Flow control: Receiver advertises a window size to prevent buffer overflow
  • Congestion control: TCP slows down when the network is congested (slow start, AIMD)
TCP 3-Way Handshake:
Client                    Server
  |  ── SYN (seq=x) ──→   |    "I want to connect"
  |  ←── SYN-ACK ─────    |    "OK, I'm ready"
  |  ── ACK ────────→     |    "Let's go"
  |                        |
  |  ── DATA ───────→     |    Now data can flow
  |  ←── ACK ─────────    |    Receiver confirms

UDP: The Fast, Unreliable Protocol

UDP (User Datagram Protocol) is deliberately minimal. It sends packets and forgets them. No handshake, no acknowledgments, no ordering guarantees. It does have checksums for corruption detection, but it drops corrupted packets silently.

Why would you ever want this? Because overhead is latency. UDP is ideal when:

  • Speed matters more than completeness (video calls — a dropped frame is better than a late frame)
  • The application handles retransmission at a higher level
  • Messages are independent and short (DNS queries — a 100-byte question, not a stream)
  • Multicast/broadcast is needed (UDP supports one-to-many; TCP does not)
Property TCP UDP
Connection Connection-oriented (handshake) Connectionless (fire-and-forget)
Reliability Guaranteed delivery + retransmit Best-effort, no retransmit
Ordering In-order delivery Out-of-order possible
Speed Slower (overhead) Faster (minimal overhead)
Use cases HTTP, SSH, file transfer, databases DNS, video/audio streaming, gaming, QUIC
Header size 20–60 bytes 8 bytes
⚠️ TCP Head-of-Line Blocking

TCP's ordered delivery creates a subtle problem: if packet #5 is lost, packets #6–100 must wait in the receive buffer until #5 is retransmitted and received. This is called head-of-line blocking and is why HTTP/2 (running over TCP) still suffers from it at the transport layer. HTTP/3 solves this by moving to QUIC (UDP-based), where stream blocking is per-stream, not connection-wide.

DNS: The Internet's Phone Book

DNS (Domain Name System) translates human-readable domain names like api.stripe.com into IP addresses like 34.149.212.105 that routers use to deliver packets. It's a massively distributed, hierarchical, eventually-consistent database — and it's often the hidden bottleneck or single point of failure in systems that don't account for it.

Excalidraw — DNS Resolution Pipeline

The DNS Resolution Chain

When you type www.stripe.com and hit Enter, here's what happens:

1

Browser & OS Cache Check

Your browser checks its own DNS cache first, then the OS cache (e.g., /etc/hosts on Linux). If found and TTL hasn't expired, done immediately — zero network hops.

2

Recursive Resolver (ISP / 8.8.8.8)

If not cached locally, the OS asks your configured recursive resolver (your ISP's DNS or a public one like Google's 8.8.8.8 or Cloudflare's 1.1.1.1). The resolver also has a cache.

3

Root Nameserver

If the resolver has no cache hit, it asks one of the 13 root nameserver clusters (like a.root-servers.net). The root doesn't know the IP — it returns the address of the TLD nameserver for .com.

4

TLD Nameserver (.com)

The TLD nameserver for .com (operated by Verisign) doesn't know the final IP either. It returns the address of Stripe's authoritative nameserver.

5

Authoritative Nameserver (Stripe's)

Stripe's authoritative nameserver (e.g., running on Route 53 or Cloudflare) has the definitive answer. It returns an A record with the IP address and a TTL. The resolver caches this for TTL seconds, then returns it to your browser.

Key DNS Record Types

RecordMapsExample
A Hostname → IPv4 address api.stripe.com → 34.149.212.105
AAAA Hostname → IPv6 address api.stripe.com → 2600:1f18::1
CNAME Alias → canonical hostname www.stripe.com → stripe.com
MX Domain → mail server stripe.com → aspmx.l.google.com
TXT Domain → arbitrary text SPF, DKIM, ownership verification
NS Domain → authoritative nameserver stripe.com → ns1.p23.dynect.net
SRV Service → host + port _http._tcp.example.com → host:port

TTL: The DNS Knob That Controls Freshness

Every DNS record has a TTL (Time to Live) in seconds. Caches hold the record until TTL expires, then re-query. This creates a system design tradeoff:

  • Short TTL (30–300s): Changes propagate quickly. Good for failover (update DNS to redirect traffic). High resolver query load and slightly higher latency.
  • Long TTL (3600–86400s): Low query load, fast resolution. Bad for failover — old IPs remain cached for hours.
⚠️ Common Interview Trap: DNS in Failover Scenarios

If you design a system that fails over by "changing the DNS record," make sure you account for TTL propagation time. If your TTL is 3600s, it can take up to an hour for all resolvers to get the new IP. In practice, you pre-lower the TTL before a planned cutover (set it to 60s a day in advance), then raise it again after the migration is stable.

HTTP: The Evolution from 1.1 to 3

HTTP is the application protocol that the web runs on. Understanding its evolution is essential because each version was designed to solve specific performance problems — and those solutions show up directly in CDN configuration, load balancer settings, and API design decisions.

HTTP/1.1 1997

The Foundation

  • Persistent connections (Keep-Alive)
  • Pipelining (rarely used — buggy)
  • Chunked transfer encoding
  • Virtual hosting (Host header)

Problem: One request at a time per connection. Browsers open 6 parallel connections per domain as a workaround. Still head-of-line blocked.

HTTP/2 2015

Multiplexing

  • Binary framing (not text)
  • Multiplexing: multiple streams over 1 TCP connection
  • Header compression (HPACK)
  • Server push (server sends resources proactively)
  • Stream prioritization

Problem: Still TCP — single dropped packet blocks all streams (transport-layer HOL blocking).

HTTP/3 2022

QUIC Transport

  • Runs on QUIC (UDP-based)
  • 0-RTT connection resumption
  • Per-stream loss recovery — no HOL blocking
  • Built-in TLS 1.3
  • Connection migration (IP changes)

Win: Mobile-friendly. Connection survives switching from WiFi to LTE without reconnecting.

HTTP/2 Multiplexing: Why It Matters

The single biggest HTTP/1.1 problem was that each response had to complete before the next request could be sent on that connection. HTTP/2 introduces the concept of streams — independent, bidirectional sequences of frames multiplexed over a single TCP connection. 

HTTP/1.1 (1 request at a time per connection):
  Request A ──→ Response A
                            Request B ──→ Response B
                                                      Request C ──→ Response C

HTTP/2 (all streams interleaved on 1 TCP connection):
  Stream 1:  ─Request─────────────Response─────
  Stream 2:    ─Request────Response─────────────
  Stream 3:      ─Request───────────────Response─
  (All happening concurrently on a single connection)

This eliminates the "6 parallel connections" hack browsers needed with HTTP/1.1 and dramatically reduces server load and connection overhead.

TLS & HTTPS: Secure Connections

TLS (Transport Layer Security) wraps TCP to provide three properties: confidentiality (encryption), integrity (tamper detection via MACs), and authenticity (server identity via certificates). HTTPS = HTTP over TLS.

TLS 1.2 Handshake (the old way — 2 RTTs)

Client                              Server
  ── ClientHello ──────────────────→   (cipher suites, random)
  ←── ServerHello + Certificate ─────  (chosen cipher, cert)
  ── ClientKeyExchange + ChangeCipher→ (encrypted pre-master)
  ←── ChangeCipherSpec + Finished ───  (both sides derive same key)

  ── Encrypted HTTP Request ──────→
  ←── Encrypted HTTP Response ───────

  Cost: 2 RTTs before first byte of data

TLS 1.3 Handshake (current — 1 RTT)

Client                              Server
  ── ClientHello + key_share ──────→  (includes DH parameters upfront)
  ←── ServerHello + key_share + cert─ (server can derive key immediately)
  ── Finished ─────────────────────→

  ── Encrypted HTTP Request ──────→   (1 RTT to first data!)
  ←── Encrypted HTTP Response ───────

TLS 1.3 also supports 0-RTT session resumption for repeated connections, where a returning client can send encrypted data in the very first packet using a previously negotiated session key. HTTP/3 (QUIC) bakes TLS 1.3 in at the transport layer.

💡 Why TLS Latency Matters for Architecture

Each TLS handshake adds 1–2 RTTs. For a US→Europe connection (~100ms RTT), that's 100–200ms of overhead before any data flows. This is exactly why connection pooling (keep connections alive and reuse them), TLS session resumption, and CDN edge nodes (terminate TLS close to the user) are so important. Moving the TLS endpoint from a datacenter 100ms away to a CDN node 5ms away saves 190ms per new connection.

HTTP Status Codes: The Vocabulary You Must Know

Status codes communicate the result of an HTTP request. In system design, they determine how clients behave: when to retry, when to cache, when to redirect. Using the wrong status code is a real API design bug.

2xx
Success
  • 200 OK
  • 201 Created
  • 204 No Content
  • 206 Partial Content
3xx
Redirect
  • 301 Moved Permanently
  • 302 Found (temp)
  • 304 Not Modified
  • 307 Temporary Redirect
4xx
Client Error
  • 400 Bad Request
  • 401 Unauthorized
  • 403 Forbidden
  • 404 Not Found
  • 409 Conflict
  • 429 Too Many Requests
5xx
Server Error
  • 500 Internal Server Error
  • 502 Bad Gateway
  • 503 Service Unavailable
  • 504 Gateway Timeout

The distinction between 401 vs 403 is subtle but important: 401 means "you're not authenticated — please log in." 403 means "you're authenticated but not authorized — you don't have permission." Using the wrong one leaks information or breaks client retry logic.

502 vs 503 vs 504: 503 means the server is overwhelmed or in maintenance (correct for rate limiting and shutdowns). 502 means a gateway received an invalid response from an upstream server. 504 means the gateway timed out waiting for upstream. These codes are what your load balancer/API gateway returns — your servers return 500 for their own errors.

Connecting Networking to System Design Decisions

Here's how today's networking concepts map directly to architecture decisions you'll make in interviews:

ConceptArchitecture Impact
TCP connection overhead Use connection pools for databases and internal services. Don't open a new connection per request.
DNS TTL Lower TTL before migrations/failovers. Use short TTLs for services that need fast failover (Route 53 health checks).
HTTP/2 multiplexing Enable HTTP/2 on API gateways. Eliminates the need for domain sharding (HTTP/1.1 workaround). Use for gRPC (always HTTP/2).
TLS termination Terminate TLS at the load balancer or CDN edge (closest to user). Backend services can communicate over HTTP internally (on trusted network).
UDP for latency-sensitive Games, video calls, IoT telemetry. Also QUIC/HTTP/3 — which CDNs increasingly support.
Head-of-line blocking Motivates HTTP/2 (application-level), HTTP/3/QUIC (transport-level). Also motivates keeping payloads small so blocking duration is minimal.
TCP 4-tuple (src IP, src port, dst IP, dst port) uniquely identifies a connection. A single server can handle ~65k concurrent connections to a single destination IP:port. For more, use connection multiplexing or multiple IPs.

Real-World Example: How Stripe handles API connections

Stripe's API serves millions of requests per day. Their architecture illustrates these principles:

  • TLS terminated at Cloudflare edge — TLS handshake happens within ~20ms for most users, regardless of where Stripe's servers are
  • HTTP/2 on their API — mobile SDK clients can pipeline multiple requests over a single connection
  • Short DNS TTL (60s) — allows fast failover between datacenters during incidents
  • Connection pooling to databases — a fleet of app servers each maintains a pool of ~20 PostgreSQL connections, rather than opening a new connection per request

✅ Key Takeaways for Day 3

  • The 4-layer TCP/IP model: Application (HTTP), Transport (TCP/UDP), Internet (IP), Network Access (Ethernet). Each layer adds/strips headers.
  • TCP guarantees reliable, ordered delivery via handshake + ACKs — use for anything where correctness matters. UDP is fast but unreliable — use for latency-sensitive, loss-tolerant workloads.
  • DNS resolves names through a hierarchy: local cache → recursive resolver → root NS → TLD NS → authoritative NS. TTL controls staleness — lower it before planned failovers.
  • HTTP/1.1 is text-based, one-request-at-a-time per connection. HTTP/2 adds binary multiplexing over one TCP connection. HTTP/3 moves to QUIC (UDP) to eliminate transport-layer HOL blocking.
  • TLS 1.3 costs 1 RTT (vs 2 RTTs for TLS 1.2). Terminate TLS at the CDN edge to minimize handshake latency for users. Reuse connections via pooling to amortize this cost.
  • Every interview answer about "reducing latency" should consider: fewer round trips (HTTP/2, connection reuse), closer endpoints (CDN, edge), and reduced DNS lookup time.

🧠 Practice Questions

1
A client reports that the first API call after a period of inactivity is always slow (~200ms), but subsequent calls in the same session are fast (~5ms). What networking concepts explain this, and how would you fix it?
Hint: Think about what happens the first time vs subsequent times — connection setup, TLS, DNS caching
2
Your team deploys an update to production by changing a DNS A record to point to new servers. An hour later, some users still see the old version. Why, and how would you architect the next migration to avoid this?
Hint: TTL — what value was it set to, and what's the propagation timeline?
3
You're designing a live video streaming platform (like Twitch). Should the video data stream use TCP or UDP? What about the chat messages? Justify both decisions.
Hint: Latency vs correctness — a frozen frame vs a dropped frame in live video has different impact than missing a chat message
4
An API returns HTTP 200 with {"error": "user not found"} in the body. What's wrong with this pattern, and what should the correct response be? Why does it matter for distributed systems?
Hint: How do load balancers, monitoring systems, and retry logic use HTTP status codes?
5
Explain why enabling HTTP/2 on your API gateway might let you reduce the number of TCP connections your mobile app maintains. How does this reduce battery usage on mobile devices?
Hint: HTTP/1.1 requires multiple connections for parallelism; each idle TCP connection keeps a radio alive on mobile

Excalidraw — System Design Interview Framework

Why You Need a Framework

System design interviews are intentionally open-ended. Unlike coding problems where there's a single correct answer, system design problems have infinite valid approaches. This is by design — interviewers are evaluating your thought process, not just your final architecture.

Without a framework, candidates make these common mistakes:

  • Jumping straight to a solution before understanding the problem
  • Designing for the wrong scale — building for Google-level traffic when the system needs to handle 10k users
  • Missing critical requirements that invalidate the entire architecture later
  • Spending 40 of 45 minutes on one component, leaving no time for other parts
  • Being too vague or too detailed in the wrong places

A framework doesn't constrain your thinking — it focuses it. It tells you what to talk about and in what order, so you can use your 45 minutes deliberately and demonstrate principal-level thinking.

💡 What Interviewers Are Actually Evaluating

Senior interviewers want to see: (1) that you can navigate ambiguity, (2) that you reason about tradeoffs explicitly, (3) that you understand the limits of your design, and (4) that you can drive a technical conversation. They are NOT looking for a textbook-perfect answer.

The 5-Step Framework

Use this process for every single system design problem, whether it's "Design Twitter" or "Design a distributed key-value store." The steps build on each other — skipping one creates problems later.

1

Clarify Requirements

⏱ Spend 5–10 minutes

Never start designing until you fully understand what you're building. Ask questions to uncover both functional requirements (what the system does) and non-functional requirements (how well it does it).

Questions to always ask:

  • Who are the users and what's the primary use case?
  • What scale are we designing for? (users, requests per second, data volume)
  • Which features are in scope vs out of scope?
  • What does "success" look like? (latency, availability, consistency requirements)
  • Are there any special constraints? (budget, team size, existing infrastructure)
2

Estimate Scale & Constraints

⏱ Spend 3–5 minutes

Do back-of-envelope calculations to understand the scale of the problem. These estimates drive every architectural decision — your storage choice, your sharding strategy, your caching tier.

Key things to estimate:

  • QPS (Queries Per Second) — peak vs average read/write ratio
  • Storage — data per record × records per day × retention period
  • Bandwidth — throughput for reads and writes
  • Memory — what needs to live in cache

Don't be precise — be right within an order of magnitude. 10^6 users × 10 writes/day = 10^7 writes/day ≈ 115 writes/sec is good enough.

3

Define the API

⏱ Spend 3–5 minutes

Before drawing boxes and arrows, define the API contract for your system. This forces you to think about what the system needs to do at a concrete level, and it anchors the rest of your design.

For each major operation, define:

  • The endpoint (method, path or function signature)
  • The inputs (parameters, request body)
  • The outputs (response format, error cases)
  • Any rate limits or constraints

Example for a URL shortener:

POST /shorten
  body: { url: string, ttl?: seconds }
  returns: { short_url: string }

GET /:code → 301 redirect to original URL
4

High-Level Architecture

⏱ Spend 10–15 minutes

Now draw the architecture. Start with the core data flow — how does a request travel through your system from client to response? Draw the boxes and arrows. Name your components clearly.

Identify and place:

  • Clients (web, mobile, API consumers)
  • Load balancers & API gateways
  • Application/service layer
  • Storage (databases, object store, cache)
  • Asynchronous components (queues, workers)
  • External integrations (CDN, email, push notifications)

Always justify your choices. "I'm using PostgreSQL here because the data is relational and we need ACID guarantees for financial records" is 10× better than just writing "DB" in a box.

5

Deep Dive & Tradeoffs

⏱ Spend 10–15 minutes

Pick 2–3 areas to go deep. The interviewer will guide you, but you should also proactively identify the hardest parts of your design and tackle them.

Common deep dive areas:

  • How does your system handle failures and partial outages?
  • How does it scale from 10k to 10M users?
  • What's the data model — show the actual schema?
  • How does the most performance-critical path work end-to-end?
  • What consistency/latency tradeoffs did you make and why?

At the principal level, you're expected to identify failure modes proactively — without being asked, say: "A weakness of this design is X. Here's how I'd address it..."

Functional vs Non-Functional Requirements

This distinction is critical. Every system design interview starts with requirements gathering, and you need to capture both types.

Functional Requirements

What the system does. These are the features and behaviors visible to users.

Example — Designing a URL Shortener

Design a URL shortening service like bit.ly.


Got it. Let me start by clarifying the requirements. Can users create short URLs? Can they customize the alias? Do links expire? Is there analytics on link clicks?


Yes to creating, no to custom aliases, links expire after 1 year, analytics are out of scope for now.


Perfect. So functional requirements: (1) Given a long URL, generate a short URL. (2) Redirect short URL to long URL. (3) URLs expire after 1 year. Does that capture it?


Yes, that's right.

Non-Functional Requirements

How the system performs. These define the quality attributes and constraints that shape your architecture.

Attribute Question to Ask Why It Matters
Availability What SLA do we need? (99.9%? 99.99%?) Determines redundancy, failover strategy
Latency What's acceptable p99 response time? Drives caching and async processing decisions
Consistency Is stale data acceptable? For how long? Impacts database choice and replication strategy
Durability Can we lose any data if a server crashes? Determines write durability, backup strategy
Scale How many users, QPS, and how much data? The most fundamental constraint on everything
Security What data is sensitive? Who can access what? Affects auth design, encryption, compliance
⚠️ Common Mistake: Assuming Requirements

Never assume the scale. A system for 1,000 users is designed completely differently from one for 1,000,000,000. Always ask, always confirm, always make your assumptions explicit.

Back-of-Envelope Estimation Basics

You don't need to be precise — you need to be roughly right about the order of magnitude. This tells you whether you need one server or one thousand, whether you need 1TB or 1PB of storage.

Worked Example — Twitter Scale Estimation

Let's say Twitter has 300M daily active users. Estimate the storage requirements for tweets.

# Given
Daily Active Users (DAU)     = 300,000,000
Tweets per user per day      = 0.5 (avg)
Bytes per tweet              = 300 bytes

# Writes per second
Total tweets/day  = 300M × 0.5 = 150M tweets/day
Tweets/second     = 150M / 86,400 ≈ 1,736 tweets/sec

# Storage per day
Daily storage     = 150M × 300 bytes = 45 GB/day

# Storage for 5 years
5-year storage    = 45 GB × 365 × 5 ≈ 82 TB

# With replication (3×)
Total storage     ≈ 250 TB

From this, we know we need to store ~250TB of tweet text over 5 years. That's significant — it tells us we'll need multiple database servers and probably some kind of archival strategy.

✓ Estimation Best Practice

Always state your assumptions explicitly. "I'm assuming 300M DAU, 0.5 tweets per user per day, 300 bytes per tweet." The interviewer can correct wrong assumptions, but can't give you credit for assumptions they don't know you're making.

Managing 45 Minutes Effectively

Most system design interviews are 45–60 minutes. Here's how to allocate that time:

⏰ 45-Minute Interview Time Budget

0–10 min

Clarify Requirements
Ask questions, establish scope, confirm non-functional requirements

10–15 min

Estimate & API
Back-of-envelope calculations, define API endpoints

15–30 min

High-Level Design
Draw the architecture, justify every component choice

30–45 min

Deep Dive
Pick 2–3 hard problems, tradeoffs, failure scenarios

Keeping the Interviewer Engaged

System design interviews are conversations, not presentations. You're expected to involve the interviewer constantly:

  • "I'm thinking of using a relational DB here. Does that work for your use case?"
  • "I see two approaches — X is simpler but doesn't scale as well; Y is more complex but handles 10× traffic. Which direction would you like me to go?"
  • "I'm going to make an assumption here that reads are 100× more frequent than writes. Does that match your expectations?"

This signals collaborative thinking — a skill that's even more valuable at the principal level than technical knowledge.

How Interviewers Score Your Answer

Most companies use a rubric with dimensions like these. Understanding the rubric helps you know where to focus your energy.

Problem Scoping (20%)

Did you ask the right questions? Did you establish clear requirements? Did you scope the problem appropriately?

System Knowledge (25%)

Do you know the components that make systems work? Load balancers, databases, caches, queues — do you understand their properties?

Tradeoff Analysis (30%)

Do you reason about tradeoffs explicitly? Do you acknowledge limitations? Do you know when to use what and why?

Design Quality (15%)

Is the design coherent, scalable, and correct? Does it actually solve the problem at the stated scale?

Communication (10%)

Are you clear and structured? Do you drive the conversation? Can you explain complex concepts simply?

Notice that Tradeoff Analysis is the highest-weighted dimension. This is why the course spends so much time on the "why" behind each architectural decision — knowing that Kafka exists is worth less than knowing when to use Kafka vs SQS vs direct DB writes and why.

Red Flags That Kill Great Candidates

Even experienced engineers fail system design interviews by making these mistakes:

🚩 Red Flag #1: Starting with the solution

The moment the interviewer says "Design YouTube," some candidates immediately say "Okay, we need S3 for video storage, Kafka for events, and a CDN." This skips requirements entirely. Always ask: who are the users, what are they doing, what's the scale?

🚩 Red Flag #2: Defending a bad design

When the interviewer probes a weak point in your design, agree and adapt. "You're right, that's a single point of failure. Let me add a replica..." Defending wrong answers is worse than making them.

🚩 Red Flag #3: Buzzword engineering

"I'd use microservices with Kubernetes and Kafka and GraphQL and Redis" without explaining why is a red flag. Every technology choice must be justified. If you can't explain the tradeoff, don't mention the technology.

🚩 Red Flag #4: Silence

Thinking quietly for 3 minutes is death in a system design interview. Think out loud. "I'm considering two options here — let me think through the tradeoffs of each..." Even wrong reasoning stated out loud beats silence.

✅ Key Takeaways for Day 1

  • Use the 5-step framework for every system design interview: Clarify → Estimate → API → Architecture → Deep Dive
  • Always ask clarifying questions before designing. Never assume scale or requirements.
  • Separate functional requirements (what it does) from non-functional requirements (how well it does it)
  • Back-of-envelope estimates drive architectural decisions — learn to do them quickly
  • Interviewers score tradeoff reasoning more heavily than technical knowledge
  • Think out loud. System design is a collaborative conversation, not a solo presentation.

🧠 Practice Questions

1
You're asked to "Design Instagram." Write down 8 clarifying questions you'd ask before drawing a single box. Then categorize them as functional vs non-functional.
Hint: Think about who posts, who reads, what media types, what features, what scale
2
Estimate the storage requirements for a WhatsApp-like system with 1 billion daily active users, where each user sends an average of 40 messages per day. Assume 75% are text (1KB each), 20% are images (200KB each), and 5% are videos (5MB each).
Hint: Calculate text, image, and video storage separately, then add a 3× replication factor
3
What's the difference between latency and throughput? Give an example where a system could have high throughput but poor latency, and another where it has low latency but poor throughput.
Hint: Think about batch processing systems vs interactive APIs
4
An interviewer asks you to design a "parking lot system." List what you'd consider in-scope vs out-of-scope, and identify the 3 most important non-functional requirements for this system.
5
Practice the time allocation. Set a 45-minute timer and try to apply the 5-step framework to "Design a file sharing service like Dropbox." Don't worry about getting it right — just practice the structure.

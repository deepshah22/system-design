Load Balancing
The Traffic Cop of Distributed Systems

Every system that survives real traffic has one thing in common: something stands between the internet and the servers, deciding who goes where. Today you learn exactly how — and why almost every system design interview answer starts by drawing this box.

Why didn't Amazon's homepage fall over on Black Friday, when 10 million people hit "Buy Now" inside the same 60 seconds?

It's not because Amazon has one giant, impossibly powerful server. It's because no single request ever talks to "the server" — it talks to a thin layer that instantly decides which of thousands of identical servers should handle it. That layer is the load balancer, and it is almost always the first box drawn in a system design interview.

Visual Memory Anchor

🚦

Picture this

A traffic cop standing at a fork in the road, waving each incoming car down a different lane so no single lane backs up — while a few lanes are coned off for repairs and quietly get no traffic at all.

The Story

Meet LB Lola. She stands at the only door into Requestville. Behind her are three identical workers — Server Sam, Server Sam-2, and Server Sam-3 — each capable of handling any request, but each with a limited number of hands.

When a request walks in, Lola doesn't think hard. She glances at who's least busy, taps them on the shoulder, and sends the request their way. If Sam-2 collapses from exhaustion (a crash), Lola simply stops sending him anyone until he's back on his feet — she noticed because she checks on each Sam every few seconds with a quick "you alive?" (a health check). Nobody outside the door ever knows Sam-2 had a bad day.

What should stay in your head
Load Balancer = One smart traffic cop spreading requests across many identical workers, so no single worker — and no single point — ever falls over.

Load Balancing Algorithms

Lola needs a rule for picking the next worker. The rule she uses changes the system's behavior dramatically:

AlgorithmHow it picksGood for
Round RobinCycles through servers in order: 1, 2, 3, 1, 2, 3…Servers with identical capacity
Weighted Round RobinSame as above, but stronger servers get more turnsMixed hardware (some servers are bigger)
Least ConnectionsSends the request to whoever has the fewest active connections right nowLong-lived or uneven requests (e.g. websockets)
IP HashHashes the client IP to always route the same client to the same serverSimple session stickiness
Consistent HashingMaps both servers and requests onto a hash ringCaches/sharded stores where minimizing re-mapping on scale-out matters

Layer 4 vs Layer 7 Load Balancing

Layer 4 (Transport)

Looks only at IP + TCP/UDP port. Doesn't read the actual HTTP request. Extremely fast, but "dumb" — it can't route based on URL path or headers. Used for raw throughput (e.g. AWS NLB).

Layer 7 (Application)

Reads the actual HTTP request — path, headers, cookies — and can route /api/* to one fleet and /images/* to another. Slightly slower (more to inspect) but far more flexible. Used for most web traffic (AWS ALB, NGINX, Envoy).

Health Checks & Sticky Sessions

Health checks: the load balancer periodically pings each backend (GET /healthz). Miss enough checks in a row and the server is pulled from rotation automatically — no human in the loop.

Sticky sessions: sometimes a user's session state lives only on one server's memory. The load balancer pins that user (via a cookie or IP hash) to the same server for the session. This is a crutch — the better fix is usually a shared session store (Redis), which you'll meet on Day 7.

The load balancer itself is a single point of failure — so production systems run at least two, in active-passive or active-active pairs, often fronted by DNS or a floating IP.

Active Recall

❓ What specific problem does a load balancer solve, in one sentence? (click to reveal)
It prevents any single server from being overwhelmed by spreading incoming requests across a fleet of servers, while routing around any server that's unhealthy.

Feynman Test

Explain a load balancer to a 10-year-old. Write it below, then compare.

"Imagine one cash register at a busy store with a huge line. Now imagine three registers open, and a helper at the front pointing each customer to whichever register is shortest. That helper is the load balancer."

Real-World Engineering

🎬 Netflix

Uses AWS Elastic Load Balancers in front of thousands of microservice instances, with Zuul as an app-layer gateway for routing and resiliency.

📦 Amazon

ALB for HTTP routing by path/host, NLB for raw TCP throughput on payment and inventory paths.

🔎 Google

Built Maglev, a custom software load balancer handling all of Google's inbound traffic with consistent hashing for failover.

🚗 Uber

Uses consistent hashing in its load balancing layer so that re-balancing on scale-out doesn't reshuffle most existing mappings.

Flashcards (Spaced Repetition)

Q: Why use a load balancer?
A: To distribute traffic across multiple servers, avoid overload, and route around failures.
Q: Round robin vs least connections — when does least connections win?
A: When requests vary in duration/cost (e.g. long-lived connections), so an even count doesn't mean even load.
Q: L4 vs L7 load balancing — what's the core tradeoff?
A: L4 is faster but routes blind (IP/port only); L7 reads the HTTP request and can route smartly, at a small latency cost.

Memory Test — Answer Without Looking Up

  • What does a load balancer do when a health check fails three times in a row?
  • Name two load balancing algorithms and one scenario where each wins.
  • Why is a single load balancer itself a risk, and how do production systems fix that?
  • What's the difference between Layer 4 and Layer 7 load balancing?
  • Why are sticky sessions considered a crutch, and what's the better long-term fix?
If you only remember one thing: a load balancer's entire job is to make a fleet of replaceable servers look, from the outside, like one perfectly reliable one.

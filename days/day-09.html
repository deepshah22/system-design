ACID Properties & Transactions
The Contract That Stops Your Money From Vanishing

Yesterday you learned Database Dave's whole personality is "consistency above all else." Today you open up the actual mechanism behind that personality — the four-letter contract every relational database enforces on every transaction, and the very real, very specific bugs it exists to prevent. By the end you'll be able to name the exact ACID letter that fails in any given failure scenario, not just recite the acronym.

You transfer $500 to a friend. The power goes out at your bank's data center at the exact millisecond after your account is debited but before your friend's account is credited. When the power comes back — where did the $500 go?

The honest, infuriating answer should be: nowhere. It should be exactly as if the transfer never started at all. Not "your $500 is debited and we're investigating," not "your friend got the money twice because the retry doubled up" — the transfer either completes completely, or it's as if you clicked nothing. That single guarantee, repeated correctly under crashes, concurrent transactions, and partial failures, is what ACID actually buys you. Most engineers can recite the acronym; principal engineers can tell you exactly which letter breaks in a given incident report.

Visual Memory Anchor

📜

Picture this

A notary's desk. Every document either gets fully signed, sealed, and filed — or it's torn up and thrown in the bin, never half-filed (Atomicity). The notary refuses to file anything that breaks the office's standing rules, like a deed naming an owner who doesn't exist (Consistency). Two clients negotiating contracts at adjacent desks can't see each other's unfinished paperwork (Isolation). And once a document is stamped "FILED," it goes into a fireproof safe, not a desk drawer that burns with the building (Durability).

The Story

Database Dave is processing a transfer: debit Amy $500, credit Bo $500. Halfway through, the power cuts. When the lights come back, Dave doesn't shrug and guess — he checks his write-ahead log, a running diary he wrote before touching either balance, and either finishes the whole transfer from where the log says it was, or rolls it back to before it started. Amy never ends up $500 poorer with Bo unpaid. That's Atomicity: all or nothing, no matter how rude the power outage is.

Dave also refuses to file a transfer that would make Bo's balance negative if Bo has an overdraft rule against it, or that would make the bank's total ledger not sum to zero — every transaction must leave the books in a state that obeys the rules Dave was given. That's Consistency — not "eventually consistent" in the NoSQL sense, but "never violates a constraint," full stop.

While Dave is mid-transfer, Database Dave's coworker starts a second transaction reading Amy's balance to print a statement. If that coworker saw Amy's balance after the debit but before Bo's credit landed, the statement would show money that's nowhere — debited from Amy, not yet at Bo. Dave's office enforces Isolation: concurrent transactions are scheduled so no one ever reads another transaction's half-finished scratch work, as if every transaction ran one at a time even though several run concurrently for speed.

Finally, the instant Dave stamps a transaction "COMMITTED," it's written to disk in a way that survives a crash, a reboot, even a fire in the building next door (replicated write-ahead log). That's Durability — "committed" is a promise, not a suggestion, and the database keeps that promise even if it dies one millisecond later.

What should stay in your head
ACID = a transaction either fully happens or fully doesn't (Atomicity), never breaks your rules (Consistency), never leaks half-finished work to another transaction (Isolation), and survives a crash once committed (Durability). Each letter is a named answer to a specific failure mode — not a vague promise of "correctness."

ACID In Depth

Atomicity — all or nothing

A transaction is a single unit: every statement inside it commits together, or none of them do. The mechanism is almost always a write-ahead log (WAL) — the database writes "I intend to do X, then Y" to a durable log before touching the actual data pages. If a crash happens mid-transaction, recovery replays the log to either finish a transaction that was fully logged as committed, or undo one that wasn't. This is also why ROLLBACK works on demand — the database just stops applying the log and discards it.

Consistency — never break the rules

This is the most overloaded word in distributed systems (it means something different in CAP theorem — Day 10 untangles that explicitly). In ACID, Consistency means a transaction can only move the database from one valid state to another valid state, honoring every constraint: foreign keys, uniqueness, check constraints, application-level invariants enforced inside the transaction. It's the database's way of saying "I will never let your bug corrupt the data, even if your application logic tries to."

Isolation — concurrent transactions don't leak

In theory, isolation means transactions behave as if they ran one after another (serializable), even though the database actually interleaves them for throughput. In practice, full serializability is expensive, so SQL defines four standard isolation levels that trade strictness for speed — covered in depth below.

Durability — committed means committed

Once a transaction returns "success," that data must survive a crash, a reboot, or a power loss. This is usually achieved by fsync-ing the write-ahead log to disk (or to multiple replicas) before acknowledging the commit — which is exactly why durability has a latency cost: waiting for a physical disk write, or for a quorum of replicas to confirm, is slower than just writing to memory and hoping.

Isolation Levels: What Each One Actually Fixes

Isolation isn't binary. SQL defines four standard levels, each one closing off one more category of "read anomaly" — and each one costs more throughput than the last.

LevelPreventsStill allows
Read UncommittedNothing extraDirty reads — you can read another transaction's uncommitted, possibly-about-to-be-rolled-back changes
Read CommittedDirty readsNon-repeatable reads — re-reading the same row twice in one transaction can return different values if someone else committed in between
Repeatable ReadDirty reads, non-repeatable readsPhantom reads — re-running the same range query can return new rows that another transaction inserted
SerializableAll of the aboveNothing — but pays the highest cost: heavy locking or conflict detection, more aborts/retries under contention

Most production systems default to Read Committed (Postgres, Oracle) or a snapshot-based variant of Repeatable Read (MySQL/InnoDB), because true Serializable isolation under high concurrency tanks throughput. The interview-relevant skill isn't memorizing the table — it's knowing that "isolation" is a dial, not a switch, and being able to argue which level a given feature actually needs.

Write skew: the anomaly Repeatable Read does not prevent

This is the isolation-level nuance that separates a textbook answer from a principal-level one. A hospital rule requires at least one doctor on call at all times; Alice and Bob are both on call. At Repeatable Read (or Postgres's snapshot-isolation flavor of it), Alice's transaction reads "2 doctors on call, safe for me to go off-call" and Bob's transaction — running concurrently, reading from its own consistent snapshot — reads the exact same thing. Both commit. Now zero doctors are on call, and the constraint was silently violated, even though neither transaction ever read the other's uncommitted data, and neither row was literally double-booked. This is write skew: two transactions read overlapping data, make disjoint writes based on what they read, and the combination violates an invariant that neither write alone would. Repeatable Read / snapshot isolation cannot catch this because each transaction, in isolation, did nothing wrong — only true Serializable isolation (or an explicit application-level lock, like SELECT ... FOR UPDATE on a shared row) closes this gap. This is precisely why "we run Repeatable Read, so we're safe from race conditions" is a dangerous half-truth.

Concrete anomaly walkthrough

AnomalyWhat actually happens
Dirty readTransaction A sets a row's value but hasn't committed. Transaction B reads that uncommitted value. A then rolls back — B just made a decision based on data that, officially, never existed.
Non-repeatable readTransaction A reads a row (balance = $100). Transaction B updates and commits that same row (balance = $80). A reads the row again in the same transaction and now sees $80 — the "repeated" read wasn't stable.
Phantom readTransaction A runs "count all orders over $50" and gets 12. Transaction B inserts a new $60 order and commits. A re-runs the exact same query and gets 13 — a row "phantom-appeared" that A never touched directly.
Write skewTwo transactions each read the same overlapping data, then each writes to a different row based on what they read — individually valid, jointly invalid. Standard Repeatable Read does not prevent this (see above).

How Isolation Is Actually Implemented: 2PL vs. MVCC

Isolation levels describe what a database promises. They don't say how. There are two fundamentally different mechanisms in production use, and knowing which one a given database uses explains almost all of its concurrency behavior.

Two-Phase Locking (2PL)

A transaction acquires a lock before touching a row (shared lock for reads, exclusive lock for writes) and holds every lock until it commits or rolls back — it never releases a lock early to acquire a new one (that's the "two-phase" part: a growing phase, then a shrinking phase, with no interleaving). This gives true serializability, but readers block writers and writers block readers, which tanks throughput under contention — a long-running read can stall an unrelated write on the same row. SQL Server's default isolation historically leaned on locking; it's simple to reason about and expensive under load.

Multi-Version Concurrency Control (MVCC)

Instead of locking rows, the database keeps multiple versions of each row (each tagged with the transaction ID that created it) and gives every transaction a consistent snapshot of the data as of when it started. Readers never block writers and writers never block readers — a reader just gets an older version instead of waiting. This is how Postgres, MySQL/InnoDB, and Oracle all implement isolation in practice, which is exactly why real-world "Repeatable Read" in these databases is stronger than the SQL standard's minimum bar (InnoDB's Repeatable Read, for example, uses gap locking to actually prevent phantom reads, going beyond what the standard requires) — but as shown above, MVCC snapshots still don't save you from write skew, because each transaction's snapshot is individually self-consistent even when the combination isn't.

Common Mistakes & Misconceptions

The misconceptionThe reality
"Repeatable Read means my transaction is fully safe from race conditions."It prevents dirty and non-repeatable reads, but not write skew — two transactions can each read valid data and write valid-looking changes that combine into a broken state. Only Serializable (or explicit row locking) closes that gap.
"Higher isolation is strictly safer, so just always use Serializable."Serializable isolation increases aborts/retries under contention and can meaningfully reduce throughput — it's a tool for specific invariant-sensitive operations, not a free default for an entire high-traffic application.
"ACID's Consistency and CAP's Consistency mean the same thing."ACID Consistency means "never violates a constraint." CAP Consistency means "every read sees the latest write" (linearizability). They are unrelated concepts that happen to share a word — Day 10 draws this line explicitly.
"A long-running transaction is fine as long as it eventually commits."Under 2PL it holds locks the whole time, blocking other transactions; under MVCC it can force the database to retain old row versions for the duration, bloating storage and slowing vacuum/garbage collection. Long transactions are a common cause of production incidents, not a neutral choice.
"The database enforces atomicity, so I don't need idempotency in my application."Atomicity guarantees one transaction is all-or-nothing — it says nothing about what happens when a client retries a request whose response was lost after the transaction already committed. That's a separate problem (see Day 9's Stripe example), and it needs its own solution.

Active Recall

❓ Which ACID letter specifically prevents a half-finished bank transfer from surviving a crash? (click to reveal)
Atomicity (enforced via the write-ahead log, which lets recovery either replay a fully-logged transaction or discard a partial one) is what prevents a half-finished transfer. Durability is related but answers a different question: once the transfer is fully committed, Durability is what guarantees that result survives a later crash — the two letters cover different moments in the transaction's life.

Feynman Test

Explain to a 10-year-old why your bank app doesn't just "save as it goes" the way a video game auto-saves. Write it below, then compare.

"A video game auto-save is fine if it sometimes loses the last ten seconds of progress — annoying, not catastrophic. A bank transfer is two separate edits (take from one person, give to another) that have to count as ONE thing. If the bank just saved as it went, a crash between the two edits would make money disappear forever. So instead, the bank writes down its plan first, does both edits, and only then says 'done' — and if anything interrupts it partway, it can always undo back to exactly where it started, like nothing happened."

Real-World Engineering

🏦 Stripe

Built idempotency keys on top of ACID transactions so that retried payment requests (network timeouts, client retries) never double-charge a customer — the database transaction guarantees a single charge is atomic, and the idempotency layer guarantees a retried request maps to that same atomic outcome.

🛒 Amazon

Uses ACID transactions in Aurora/RDS for order placement and inventory decrement together — a customer must never be charged for an item that simultaneously fails to decrement stock, which is exactly an atomicity guarantee across two writes.

✈️ Airline booking systems

Rely on Serializable or near-Serializable isolation for seat assignment — without it, two travelers could both "successfully" book the last seat, a classic phantom-read/race-condition bug that strict isolation exists to prevent.

📊 PostgreSQL internals

Implements isolation via MVCC (Multi-Version Concurrency Control) — readers see a consistent snapshot of the data instead of blocking writers, which is how Postgres gets strong isolation without grinding concurrent throughput to a halt.

Flashcards (Spaced Repetition)

Q: What does each ACID letter guarantee, in one phrase?
A: Atomicity = all-or-nothing. Consistency = never violates constraints. Isolation = concurrent transactions don't see each other's half-finished work. Durability = survives a crash once committed.
Q: What mechanism typically implements Atomicity and Durability together?
A: The write-ahead log (WAL) — changes are logged durably before being applied, so recovery can replay or discard a transaction after a crash.
Q: Name the four SQL isolation levels in order of strictness, and one anomaly each one newly prevents.
A: Read Uncommitted (none) → Read Committed (prevents dirty reads) → Repeatable Read (prevents non-repeatable reads) → Serializable (prevents phantom reads too).
Q: Why don't most production databases default to full Serializable isolation?
A: Serializable isolation requires heavy locking or conflict detection, which reduces concurrency/throughput — most systems pick Read Committed or snapshot-based Repeatable Read as the practical default.
Q: What is write skew, and why doesn't Repeatable Read prevent it?
A: Two transactions each read the same overlapping data and write to different rows based on it — each write is individually valid but the combination violates an invariant. Repeatable Read gives each transaction a self-consistent snapshot, but doesn't detect conflicts across transactions' disjoint writes; only Serializable isolation (or explicit locking) does.
Q: What's the core difference between Two-Phase Locking (2PL) and MVCC?
A: 2PL acquires and holds locks for the transaction's duration — readers and writers block each other. MVCC keeps multiple row versions and gives each transaction a consistent snapshot — readers never block writers and vice versa, at the cost of extra storage/version bookkeeping. Postgres, MySQL/InnoDB, and Oracle all use MVCC in practice.

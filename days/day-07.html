Redis & Memcached
Two Chairs for the Same Living Room

Yesterday you learned that a cache sits between your app and your database. Today you learn what's actually sitting in that spot — and why "just use Redis" and "just use Memcached" are both right, depending entirely on what you're asking the cache to remember.

Visual Memory Anchor

🪑

Picture this

Two chairs in the same living room. One is a simple, sturdy stool — sit down, get up, nothing fancy (Memcached: key → blob). The other is a recliner with built-in drawers, a cup holder, and a footrest that folds out on command (Redis: key → rich structures you can manipulate in place).

The Story

Cache Cathy has been running her front desk for a while now, and the warehouse owner just asked her to hire some help. Two candidates show up.

The first, Mem, is fast, simple, and a little forgetful — give him a label and a box, and he'll hand the box back instantly when you ask for that label again. He doesn't care what's in the box. If the building loses power, his desk is wiped clean and he shrugs: "that's the deal, I never promised to remember forever." But because he doesn't track anything fancy, he can run on every spare desk in the building at once (multi-threaded), and the more desks Cathy gives him, the faster he gets.

The second, Red, is more particular. He doesn't just hold boxes — he organizes what's inside them. Ask him for "the top 10 trending hashtags" and he can actually maintain that ranking for you, live, because he understands sorted sets. He also keeps a notebook of everything he's done (the AOF log) and takes periodic photographs of his whole desk (RDB snapshots), so if the power goes out, he can mostly reconstruct what was there. He works alone at his own desk, one task at a time (single-threaded core loop) — but he's so fast at each task that it rarely matters, and when one desk isn't enough, Cathy gives him a team of desks that split the work between them (Redis Cluster).

Cathy doesn't pick a favorite. For session tokens and simple lookups, she calls Mem. For leaderboards, rate limiters, and anything that needs real structure, she calls Red.

What should stay in your head
Memcached = a fast, simple, multi-threaded key → blob store with no persistence. Redis = a fast, single-threaded key → data structure server with optional persistence, replication, and clustering.

Redis In Depth

Data structures

This is Redis's whole identity: every value has a type, and each type has commands that operate on it directly, in place, on the server. A list gives you LPUSH/RPOP for queues. A set gives you union and intersection — instant "mutual friends" queries. A sorted set (ZSET) keeps members ordered by score, so ZRANGE gives you a leaderboard for free. A hash lets you store and update individual fields of an object (like a user profile) without serializing the whole thing.

Persistence: RDB vs AOF

RDB (Redis Database) takes a point-in-time binary snapshot of the entire dataset on a schedule — compact, fast to restart from, but you can lose everything since the last snapshot. AOF (Append-Only File) logs every write operation as it happens — far less data loss on crash, but a larger file and slower restarts since Redis replays the whole log. Most production setups use both: RDB for fast recovery, AOF for durability between snapshots.

Redis Cluster: sharding

Redis Cluster splits the keyspace into 16,384 hash slots. Every key is hashed to a slot, and each slot is owned by one node (with replicas for failover). Clients can be redirected (MOVED) to the right node, so the cluster scales horizontally without a single coordinator becoming the bottleneck — a direct, concrete cousin of the consistent hashing you'll formalize on Day 14.

Pub/Sub

Redis lets clients SUBSCRIBE to named channels and PUBLISH messages to them; every subscriber gets a copy instantly. It's not a durable queue — if no one's listening, the message is gone — but it's the simplest possible way to fan out real-time events, and it's the conceptual bridge to the dedicated message queues (Day 18) you'll meet when you need delivery guarantees Redis pub/sub doesn't provide.

Active Recall

❓ In one sentence, what's the core architectural difference between Redis and Memcached? (click to reveal)
Memcached stores plain byte blobs behind a key with no persistence or built-in structure; Redis stores typed data structures (lists, sets, hashes, sorted sets) that the server can manipulate directly, and can optionally persist and replicate them.

Real-World Engineering

🐦 Twitter

Uses Redis heavily for timelines and the social graph — sorted sets are a natural fit for ranked, time-ordered tweet IDs.

📸 Instagram

Has used Memcached extensively for its feed and session caching, valuing its simplicity and raw throughput at massive scale.

🐙 GitHub

Uses Redis for background job queues (via Sidekiq/Resque-style workers) and caching, leaning on its data structures beyond simple key-value lookups.

📚 Stack Overflow

Has documented using Redis for caching and rate limiting, citing its low latency and rich command set as decisive over a plain key-value store.

Flashcards (Spaced Repetition)

Q: What's the core difference between Redis and Memcached?
A: Memcached stores plain blobs with no persistence; Redis stores typed data structures and can persist, replicate, and cluster them.
Q: RDB vs AOF — what's the tradeoff?
A: RDB is a compact periodic snapshot (fast restart, more data loss risk); AOF logs every write (less data loss, slower restarts).
Q: How does Redis Cluster shard data?
A: It splits the keyspace into 16,384 hash slots, each owned by a node, so the cluster scales horizontally without one coordinator.
Q: Why is Memcached often faster for raw, simple throughput?
A: It's multi-threaded with no structure to maintain, so it scales with CPU cores; Redis runs its core logic on a single thread.

Memory Test — Answer Without Looking Up

  • Name three data structures Redis supports that Memcached has no concept of.
  • What's the difference between RDB snapshots and AOF logging?
  • Why is Redis's single-threaded core not the bottleneck you'd expect?
  • How many hash slots does Redis Cluster use to shard keys, and why does that matter?
  • Give one use case better suited to Memcached and one better suited to Redis.
If you only remember one thing: Memcached is a fast, disposable key-blob store; Redis is a fast key-structure server that can also persist, replicate, and message. Choose based on whether you need the cache to understand the shape of your data — not on benchmarks alone.

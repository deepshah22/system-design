REST API Design
One Small, Guessable Front Door

For two weeks you built a fortress for your data — replicated, sharded, hashed onto a ring so it survives failure and scales. But a fortress with no door is useless. Today you build the door: an API. And REST is the discipline that keeps that door from turning into 400 mismatched entrances. Expose your system as resources, act on them with a fixed handful of verbs, keep the server stateless, and report outcomes with status codes. Learn it once, and every REST API on Earth becomes readable at a glance.

Visual Memory Anchor

🍽️🧑‍🍳

Picture this

Walk into a well-run restaurant. There's a menu — a list of nouns: the steak, the soup, table 12. Those are your resources. A waiter serves you, and here's the trick: the waiter only knows about five gestures — bring me (GET), add a new order (POST), replace my whole order (PUT), tweak one thing (PATCH), and cancel (DELETE). You never reach into the kitchen yourself; you point at a noun on the menu and make one of the five gestures. The kitchen is stateless — the waiter carries everything the cook needs on the ticket, so any cook can pick it up; nobody has to remember "oh, that's the regular who always wants it well-done." And every time you ask for something, you get a standard reply: "here you go" (200), "made it" (201), "we don't have that dish" (404), "the kitchen's on fire" (500). A menu of nouns, a waiter with five gestures, a kitchen that remembers nothing, and standard replies — that single scene is REST.

The Story

Database Dave had finally tamed his data. After Days 12–14 it was replicated, sharded, and sitting on a consistent-hash ring that could survive a dead machine without breaking a sweat. So the mobile team came knocking: "Great — how do we read and write all this?" Dave shrugged and did what felt natural: he wrote one URL per thing anyone asked for. /getUser. /getUserById. /fetchUserEmail. /updateUserName. /deleteUserAccountForever. Six months later there were over 400 endpoints, no two named alike, half of them GETs that quietly changed data, and a mobile app that broke every time someone renamed one.

A senior engineer looked at the sprawl and asked one question: "How many things does your system actually have?" Dave counted. Users, orders, products, reviews. "Four nouns," she said. "And how many things can you do to a thing? Read it, create it, replace it, tweak it, delete it. Five verbs — and HTTP already ships with all five." She drew it out: GET /users/42 to read, POST /users to create, PUT /users/42 to replace, PATCH /users/42 to tweak, DELETE /users/42 to remove. "Your 400 endpoints are really just four nouns crossed with five verbs. Everything else is noise." Dave stared. The whole mess collapsed into a table you could hold in your head.

He rebuilt the API around resources and verbs, and two things he didn't expect fell out for free. First, the server got stateless: because each request carried who you were (a token) and what you wanted (verb + URL + body), any server behind the load balancer could handle any request — so scaling out (Day 2) was suddenly trivial. Second, the mobile team stopped filing bugs, because they could now guess the API: if /users/42 reads a user, then /orders/99 reads an order — no docs needed. The lesson that stuck: a good API isn't a pile of functions you expose. It's a small, consistent vocabulary other people can predict. That predictability is the entire point of REST.

What should stay in your head
REST is four ideas working together. (1) Resources are nouns with addresses — /users/42, not /getUser?id=42. (2) A fixed set of HTTP verbs acts on them — GET reads, POST creates, PUT replaces, PATCH tweaks, DELETE removes — so behavior is guessable instead of invented per-endpoint. (3) The server is stateless: every request carries everything needed to serve it, so any server can handle any request and you scale by adding boxes. (4) Status codes report the outcome — 2xx worked, 4xx you messed up, 5xx we messed up. Nouns + verbs + statelessness + status codes = an API strangers can predict without reading your docs.

The Four REST Constraints (What "RESTful" Actually Means)

        
REST — REpresentational State Transfer — was
        described by Roy Fielding in 2000 as a set of constraints, not a library. You don't install
        REST; you obey it. Four of those constraints carry almost all the practical weight:

        

          
1. Resource-based.Everything the API exposes is a
          resource — a noun with a URL. /users is the collection; /users/42 is
          one member; /users/42/orders is that user's orders. The URL names what you're
          talking about; it should never contain a verb like /getUser or /deleteOrder.

          
2. Uniform interface.Every resource is manipulated with the
          same small set of HTTP methods, and they mean the same thing everywhere. GET on
          /users/42 and GET on /orders/9 both mean "read this" — no per-endpoint
          surprises. This uniformity is what lets a client guess your API.

          
3. Stateless.The server keeps no memory of
          previous requests. Each request carries its own context (auth token, parameters, body). Two calls in a
          row can land on two different servers and both work — which is exactly why REST scales horizontally so
          cleanly (Day 2) and rides behind a load balancer (Day 5) without sticky sessions.

          
4. Client–server + cacheable.Client and server evolve
          independently as long as the contract holds, and responses declare whether they're
          cacheable (via headers). A GET can be cached at a CDN (Day 4) or browser; a POST generally
          can't. Safe, cacheable reads are a big reason REST feels fast on the web.

        

        
There's a fifth idea — HATEOAS (responses include links to the next available
        actions, so the client discovers the API by following links) — that's the "purest" form of REST but is
        rare in practice. Most production "REST" APIs are really REST-ish: resources + verbs +
        statelessness + status codes, without full hypermedia. That's a perfectly respectable, industry-standard
        target — and it's what we mean by REST for the rest of this course.

      


      

        
Verbs, Resources & Status Codes


        
The verbs — and two words that decide everything: safe and idempotent

        
Two properties tell you almost everything about how a verb behaves under retries and caching. A verb is
        safe if it doesn't change server state (a pure read). A verb is idempotent
        if doing it once and doing it ten times leave the server in the same state.
        These aren't trivia — they're the rules that let a flaky network retry a request without corrupting your
        data (a theme we'll hammer on again in Idempotency, Day 24).

        

          
VerbMeansSafe?Idempotent?Example

          
GETRead a resource✅ yes✅ yesGET /users/42

          
POSTCreate a new resource in a collection❌ no❌ noPOST /users

          
PUTReplace a resource entirely (or create at a known id)❌ no✅ yesPUT /users/42

          
PATCHPartially update a resource❌ no➖ usually notPATCH /users/42

          
DELETERemove a resource❌ no✅ yesDELETE /users/42

        

        
The crucial pair: POST is not idempotent — send it twice and
        you create two users — which is why a payment "Submit" button double-tapped can charge you twice.
        PUT and DELETE are idempotent —
        replacing user 42 with the same data twice, or deleting it twice, ends in the same place — so a client can
        safely retry them after a timeout. That single distinction shapes how you design retries, buttons, and
        queues later in the course.


        
Status codes — the server's standard replies

        
The first digit tells you the category, and one rule cuts through all of it:
        4xx means the caller's fault, 5xx means the server's fault. That boundary decides who
        gets paged at 3 a.m.

        

          
CodeNameWhen to use it

          
200OKA successful GET/PUT/PATCH — here's your data.

          
201CreatedA POST made a new resource. Return it (and a Location header pointing at it).

          
204No ContentSuccess, nothing to return — classic for DELETE.

          
301/302Moved / FoundThe resource lives at a different URL now (redirect).

          
304Not ModifiedYour cached copy is still fresh — save the bytes (ties into caching, Day 6).

          
400Bad RequestThe request itself is malformed — broken JSON, missing required field.

          
401UnauthorizedYou're not authenticated — we don't know who you are (Auth, Day 29).

          
403ForbiddenWe know who you are, and you're not allowed to do this.

          
404Not FoundNo such resource at this URL.

          
409ConflictThe request clashes with current state — e.g. that email is taken.

          
422UnprocessableWell-formed but semantically invalid — right shape, wrong values.

          
429Too Many RequestsYou hit the rate limit — slow down (Rate Limiting, Day 24).

          
500Internal Server ErrorOur code blew up. This is the one that pages an engineer.

          
503Service UnavailableServer is overloaded or down for maintenance — try again later.

        

      


      

        
Design Rules That Pay Off Later

        
A REST API is a contract other teams build on. These conventions are what separate an API
        people enjoy from one they curse. None is mandatory — but skip them and you'll pay in support tickets.

        

          
Name collections as plural nouns./users,
          /orders — not /user, /getUsers, or /user_list.
          Nest to show relationships: /users/42/orders. Consistency is worth more than cleverness.

          
Version from day one.Put /v1/ in the path (or
          a header). The day you need a breaking change, /v2/ lets old clients keep working while new
          ones move — no flag day, no fleet-wide outage.

          
Paginate every list.Never return "all users." Use
          ?limit=50&cursor=… (cursor beats offset at scale). An unbounded list is a
          latency bomb and a memory bomb waiting for your data to grow.

          
Filter, sort, and search in the query string.The path names
          the resource; the query refines it: /orders?status=shipped&sort=-createdAt.
          Don't invent a new endpoint for every filter.

          
Return the created resource on POST.Respond
          201 with the new object (including its server-assigned id) so the client
          doesn't need a second round-trip to learn what it just made.

          
Make errors consistent and machine-readable.Every error
          should share one shape: a stable code, a human message, and maybe a
          details list. Clients handle errors by code, not by parsing prose.

          
Support safe retries with idempotency keys.For POSTs that
          must not double-fire (payments!), let the client send an Idempotency-Key header; the server
          remembers it and returns the first result on a retry. This is the production fix for POST's
          non-idempotence — and a full lesson later (Day 24).

        

      


      

        
REST vs the Alternatives (It's Not the Only Door)

        

          

            
REST over HTTP

            
Resources + verbs + status codes, usually with JSON. Universally understood, cache-friendly,
            debuggable with a browser or curl. The safe default for public and CRUD-style APIs.

            
Great for: public APIs, resource CRUD, web + mobile backends. Weakness: over/under-fetching, many round-trips.

          

          

            
GraphQL

            
One endpoint; the client writes a query describing exactly the fields it wants across
            related objects, in one round-trip. Kills over- and under-fetching — at the cost of caching being
            harder and the server doing more work to guard against expensive queries.

            
Great for: rich, nested UIs with many clients. Coming up: Day 16.

          

          

            
gRPC + Protocol Buffers

            
Binary, contract-first RPC over HTTP/2. Very fast, strongly typed, great for streaming and
            service-to-service calls inside a datacenter — but not human-readable and awkward from a
            browser.

            
Great for: internal microservice communication, low-latency streaming. Coming up: Day 17.

          

          

            
Plain RPC / RPC-style HTTP

            
Expose functions directly: POST /createUserAndSendEmail. Sometimes the most
            honest fit for an action that isn't really a noun (/search, /transfer) — but
            let it spread and you're back to Dave's 400 endpoints.

            
Great for: genuine actions/verbs that don't map to a resource. Weakness: no uniformity, easy to sprawl.

          

        

        
The through-line: they all answer "how does the outside world talk to my system?" REST wins on
        predictability and reach; GraphQL wins on flexible fetching; gRPC wins on internal
        speed and typing. Most real companies use all three — REST at the public edge,
        gRPC between internal services, GraphQL where a UI needs flexible data. Choosing is a tradeoff, never a
        winner-take-all.

Common Mistakes & Misconceptions

        

          
The misconceptionThe reality

          
"Put the action in the URL — /getUser, /users/42/delete."The URL names the resource (a noun); the verb is the HTTP method. It's GET /users/42 and DELETE /users/42. Verbs in the path are the #1 tell of a non-RESTful API.

          
"GET can change data if it's convenient."GET must be safe — read-only. Browsers, proxies, and CDNs pre-fetch and cache GETs freely; a GET that deletes data will get triggered by a crawler and ruin your day. Anything that mutates uses POST/PUT/PATCH/DELETE.

          
"Return 200 OK and put the error inside the body."Then every client must parse your body to know if it failed, and caches/monitoring see "success." Use the status code to signal outcome — 4xx for caller errors, 5xx for server errors — and put detail in the body.

          
"POST and PUT are basically the same."POST creates (server picks the id) and is not idempotent — twice makes two. PUT replaces at a known id and is idempotent — twice is the same as once. That difference decides whether a retry is safe.

          
"REST means CRUD on a database, one endpoint per table."Resources are a conceptual model, not your table layout. A resource can span several tables or none; some resources are actions or computations. Don't leak your schema through the API.

          
"Versioning is premature — I'll add it when I need it."By the time you "need" it, you have live clients that break on any change. /v1/ from day one costs nothing and buys you the ability to ever evolve safely.

        

      


      

        
Active Recall

        

          
❓ A user taps "Pay $50" on your checkout screen. The network is slow, so they tap it again. Your endpoint is POST /payments. What just went wrong, and what are two ways to make sure they're only charged once? (click to reveal)

          
The problem is that POST is not idempotent: each tap is a brand-new "create a payment" request, so two taps create two payments and the customer is charged $50 twice. Fixes: (1) Idempotency keys — the client generates one unique key for the checkout attempt and sends it as an Idempotency-Key header on every retry; the server records the key with the first result and, on any request carrying a key it has already seen, returns that same stored result instead of charging again. This is the standard production fix (it's exactly how Stripe's API works). (2) Design the write to be idempotent — e.g. model it as PUT /payments/{clientGeneratedId} so re-sending the same id just re-asserts the same payment rather than creating a new one. Either way, the fragile piece was assuming the network delivers each request exactly once — it doesn't, so safe retries have to be designed in. (Full treatment on Day 24.)

        

      


      

        
Real-World Engineering

        

          
💳 Stripe
Widely held up as the gold standard for REST design: clean plural-noun resources (/v1/charges, /v1/customers), correct status codes, and first-class idempotency keys on every mutating request so a retried payment never double-charges. Their docs are practically a REST style guide.

          
🐙 GitHub
The GitHub REST API (/repos/{owner}/{repo}/issues) is a textbook example of nested resources, cursor/link-header pagination, conditional GETs with ETags (304 Not Modified to save bandwidth), and clear /v3-style versioning across a huge public surface.

          
📞 Twilio
Twilio exposes telephony as resources — POST /Messages to send an SMS, GET /Calls to list calls — proving REST models actions (send a text) cleanly as "create a message resource," not as an RPC verb in the URL.

          
🌐 Google & the wider web
Google's public APIs follow a published REST design guide (resource names, standard methods, consistent errors). Because REST rides on plain HTTP, the same request works from a browser, curl, a CDN cache, or a mobile app — the reach that made REST the default language of web APIs.

        

      


      

        
Flashcards (Spaced Repetition)

        

          
Q: In one sentence, what is REST?

          
A: An architectural style that exposes a system as resources (nouns with URLs) acted on by a fixed, uniform set of HTTP verbs, served by a stateless server that reports outcomes with status codes — so the API is consistent and guessable.

          
        

        

          
Q: What do "safe" and "idempotent" mean, and which verbs are which?

          
A: Safe = doesn't change state (pure read). Idempotent = doing it N times equals doing it once. GET is safe + idempotent. PUT and DELETE are idempotent (not safe). POST is neither — twice creates two. PATCH is usually not idempotent.

          
        

        

          
Q: What's the one-line rule for 4xx vs 5xx status codes?

          
A: 4xx = the caller's fault (bad request, unauthorized, not found — the client should fix the request). 5xx = the server's fault (our code broke or is overloaded — the client can retry later). This decides who gets paged.

          
        

        

          
Q: Why does statelessness make REST scale so well?

          
A: Because each request carries everything needed to serve it (auth, params, body), any server can handle any request. No server-side session means you can add servers freely behind a load balancer, and a request landing on a different box each time still works.

          
        

        

          
Q: Why put a verb in the URL (/users/42/delete) is wrong?

          
A: The URL should name the resource (a noun); the action is the HTTP method. Correct is DELETE /users/42. Verbs in the path break the uniform interface, make the API unguessable, and lead straight back to hundreds of one-off endpoints.

          If you only remember one thing: a REST API is a small, guessable vocabulary, not a pile
          of functions. Model your system as resources (nouns with URLs), act on them with a fixed set of
          HTTP verbs that mean the same thing everywhere, keep the server stateless so it scales
          by adding boxes, and report every outcome with the right status code. Get that right and
          strangers can use your API without reading the docs — which is the whole point of a front door. Next up
          (Day 16) we open a second door into the same building: GraphQL, where the client, not
          the server, decides exactly what data comes back.

Data Partitioning & Sharding
When One Copy Is Too Big for One Machine

Yesterday you made a full copy of your data survive a failure. But replication has a hard ceiling: every replica still holds the entire dataset, so when the dataset outgrows a single machine's disk, RAM, or write throughput, copying it more times solves nothing. The only way forward is to cut the data into pieces and put each piece on a different machine. That cut looks simple and is quietly one of the most consequential decisions in all of system design — because the instant you split the data, you have to answer, on every single request, "which piece holds this?"

Picture this

One enormous filing cabinet has grown so full the drawers won't shut. So you buy three smaller cabinets and split the files: cabinet A holds surnames A–H, cabinet B holds I–P, cabinet C holds Q–Z. Now no single cabinet is overloaded — but a new rule appears that governs everything: before you can file or fetch a single folder, you must first know which cabinet it lives in. Get that routing wrong and you'll search the wrong cabinet forever. Picture a clerk holding a folder, pausing at the mouth of three cabinets, asking "which one?" — that pause is the heart of sharding. The cut is easy; knowing where each piece went is the whole job.

The Story

Database Dave has a well-indexed, well-replicated ledger, and business is booming. Too well. The ledger is now 4 terabytes and growing a hundred gigabytes a week. His biggest server maxes out at 2 TB of fast storage, and even the writes are backing up — a single machine can only accept so many transactions per second no matter how much he replicates. Replication gave him three identical copies, but three copies of "too big" is still too big. Adding replicas doesn't add capacity; it just re-copies the same overflowing cabinet.

So Dave does the only thing that actually helps: he splits the data itself. He takes his customer accounts and puts A–H on machine 1, I–P on machine 2, Q–Z on machine 3. Each machine now holds a third of the data and takes a third of the writes. The dataset can grow to three machines' worth of capacity — and if he needs more, he adds a fourth machine. This is sharding: horizontal partitioning of one logical dataset across many physical nodes. For the first time, capacity scales by adding machines rather than buying a bigger one.

Then two things go wrong. First, Dave notices machine 3 is on fire while machine 1 sits idle — it turns out a third of his customers are named "Singh" and "Smith," so the Q–Z cabinet is slammed. He's created a hot shard. Second, when he adds a fourth machine to relieve the pressure, he discovers that re-drawing the boundaries means physically moving a huge fraction of his data across the network while the system is live — and every request that arrives mid-move has to know the new map. Dave has just met the two demons that haunt every sharded system: uneven load and rebalancing cost. Today is about choosing a partitioning scheme that keeps both in check.

What should stay in your head
Replication copies the whole dataset for availability and read scale; partitioning (sharding) splits the dataset so no single machine holds it all — that's how you scale capacity and write throughput. Every partitioning scheme is an answer to one question — "given a key, which shard holds it?" — and is judged on three tensions: balance (is load spread evenly, or is there a hot shard?), rebalancing cost (when you add a node, how much data has to move?), and query routing (can you find a row without asking every shard?). Real systems replicate and partition: each shard is itself a small replicated cluster.

How Partitioning Actually Works

Partitioning is not replication — and you need both

It's worth nailing the difference because the two are constantly confused. Replication makes N copies of the same data — it buys availability and read throughput, but every copy is the full dataset, so it does nothing for capacity. Partitioning splits one dataset into disjoint pieces on different machines — it buys capacity and write throughput, but a single copy of each piece is now a single point of failure. So real systems do both: the data is cut into shards, and each shard is itself replicated (a leader plus followers, exactly like Day 12). A cluster of 6 machines might be 3 shards × 2 replicas. Partitioning and replication are two independent axes, and production systems turn both dials at once.

The partition key is the most important decision you'll make

Everything hinges on the partition key (also called the shard key) — the field you feed into the function that decides which shard a row lives on. Choose user_id and all of one user's data sits together on one shard (great for "load this user's profile"). Choose timestamp and all of today's data lands on one shard (great for time-range scans, terrible for write load — every new write hammers the same "newest" shard). The partition key silently determines which queries are cheap and which become expensive scatter-gather across every shard. It is very hard to change later, because changing it means re-partitioning the entire dataset. Pick it for your access patterns, not your data model.

Routing: how a request finds its shard

Once data is split, every request must be routed to the right shard. There are three common places to put that routing logic:

Client-side (smart client).The client library knows the shard map and computes the target itself. Fewest network hops, but every client must carry and refresh the map. (Cassandra drivers work this way.)
Routing tier (proxy).A dedicated proxy sits in front and forwards each request to the right shard. Clients stay dumb; the proxy is the one place that knows the map — at the cost of an extra hop. (Vitess, Twemproxy, most managed shards.)
Node-forwarding (request any node).Send the request to any node; if it doesn't own the key, it forwards to the one that does. Simple for clients, but adds internal hops. (Cassandra's coordinator pattern.)

Whatever the location, they all rely on one shared source of truth for "who owns what," and keeping that map consistent as shards move is where a lot of real-world complexity (and a lot of Day 14) lives.

Range vs Hash vs Directory (How Do You Cut?)

Range partitioning

Assign contiguous key ranges to shards (A–H, I–P, Q–Z; or Jan, Feb, Mar). Range scans are cheap — "all users M–N" hits one shard. But contiguous keys cluster, so you get hot shards (everyone signs up "today"; half the surnames start with S).

Great for: time-series, range queries. Risk: hotspots. HBase, Bigtable, Spanner.

Hash partitioning

Run the key through a hash and assign by the result (hash(key) mod N). Spreads keys evenly, killing most hotspots — but destroys ordering, so a range query becomes a scatter-gather across every shard, and naive mod N reshuffles almost everything when N changes.

Great for: even load, point lookups. Risk: no range scans. DynamoDB, Cassandra.

Directory / lookup table

Keep an explicit map from key (or key-group) to shard in a lookup service. Maximum flexibility — you can move any group anywhere and split hotspots by hand — at the cost of an extra lookup and a map that must not become a bottleneck or single point of failure.

Great for: flexibility, manual hotspot fixes. Risk: the directory itself. Vitess, many custom shards.

The hot-shard problem — the one that bites everyone

Even a perfect hash can't save you from a single hot key. If Taylor Swift's account gets a million reads a second, hashing her user_id still lands all million on one shard — hashing spreads different keys evenly, but it can't split one key across machines. The real fixes are application-level: add a random suffix to fan a hot key across several sub-keys (celebrity#1…celebrity#16) and merge on read, or cache the hot key hard in front of the shard (Day 6). A subtler version is the hot range: partition by timestamp and every write targets "now," so one shard absorbs 100% of writes while the rest idle — which is why write-heavy systems almost never range-partition on time alone. The lesson: uniform data distribution is not uniform load distribution. You have to design for the traffic, not just the keys.

The Rebalancing Problem — and Why Day 14 Exists

Your dataset grows, so you add a shard. Now the map has to change — and the terrifying question is how much data must physically move across the network while the system keeps serving traffic. This is where the naive scheme falls apart:

Naive hash(key) mod N.Change N from 4 to 5 and the modulus changes for almost every key — roughly 80% of your data has to move to a new home at once. It's a catastrophic, cluster-wide reshuffle. This is the single biggest reason nobody uses plain mod N in production.
Fixed partitions (more partitions than nodes).Pre-split into, say, 1000 partitions up front and hand out partitions to nodes. Adding a node just moves a few whole partitions to it — a bounded fraction, not the whole cluster. Elasticsearch and Riak work this way.
Consistent hashing (tomorrow, Day 14).Arrange shards on a ring so adding one only steals keys from its immediate neighbor — on average only 1/N of the data moves, not N-1/N. This is the elegant fix that makes elastic, always-on sharding practical, and it's exactly where we go next.

The general principle: a good partitioning scheme makes rebalancing proportional — add 10% more capacity, move roughly 10% of the data — and lets it happen gradually in the background without downtime. When you see "80% of keys moved" in the simulator above under hash + add-a-shard, you are looking at precisely the problem consistent hashing was invented to solve.

Common Mistakes & Misconceptions

The misconceptionThe reality
"Adding read replicas will fix our capacity problem."Replicas copy the whole dataset, so they add read throughput but zero capacity. When the data itself is too big or writes are the bottleneck, only partitioning helps.
"A good hash function means perfectly balanced load."Hashing balances keys, not traffic. One hot key (a celebrity, a viral post) still lands entirely on one shard. Even data distribution ≠ even load distribution.
"We'll just pick a shard key later."The partition key decides which queries are cheap forever and is brutally expensive to change (it means re-partitioning everything). Choose it up front, for your access patterns.
"Sharding by hash(key) mod N is fine."It's balanced until you resize: going from N to N+1 shards moves ~(N-1)/N of all keys at once. Use fixed partitions or consistent hashing so rebalancing is proportional.
"Once sharded, every query is fast."Only queries that include the partition key are cheap. Any query that doesn't (or spans ranges under hash partitioning) becomes a scatter-gather across all shards — often slower than before.
"Cross-shard joins and transactions work like they always did."They don't. A transaction spanning shards needs distributed coordination (2PC / Saga, Days 33–34), and cross-shard joins usually have to be avoided by design or denormalized away.

Active Recall

❓ You shard a social app by hash(user_id) mod N across 4 machines and load is beautifully even. Traffic doubles, so you add a 5th machine. Two things immediately go wrong — what are they, and what would you have done differently? (click to reveal)
First, changing N from 4 to 5 changes mod N for almost every key (~80%), so the cluster has to physically move most of its data across the network at once — a massive, disruptive reshuffle while live. Second, hash partitioning means any query that isn't a point lookup by user_id — e.g. "all users who signed up this week" — is a scatter-gather across all 5 shards, so it doesn't get faster with more machines, it gets slower. The fix: don't use naive mod N. Use consistent hashing (Day 14) or a fixed large number of partitions spread over nodes, so adding a machine moves only ~1/N of the data; and choose the partition key to match your real access patterns so the important queries stay single-shard.

Real-World Engineering

🎥 YouTube / Vitess

YouTube outgrew a single MySQL and built Vitess — a sharding layer that presents many MySQL shards as one logical database, with a routing tier (VTGate) that parses queries and sends them to the right shard. It's now the open-source standard for sharding MySQL at scale.

📄 MongoDB

Ships sharding as a first-class feature: you choose a shard key, and a mongos router splits data into chunks and moves them between shards automatically as they grow. Its docs are one long warning that a bad shard key (monotonic, low-cardinality) creates hot shards.

🪶 Cassandra / DynamoDB

Both hash-partition by a partition key onto a ring (consistent hashing) so nodes join and leave with minimal data movement. DynamoDB's "hot partition" throttling is the hot-shard problem surfaced as a billing/limits concern you design around.

🔎 Elasticsearch

Splits an index into a fixed number of primary shards chosen at creation (you can't easily change it later — a real design constraint), each replicated. Adding nodes rebalances whole shards across the cluster, the fixed-partition approach in practice.

Flashcards (Spaced Repetition)

Q: What problem does partitioning solve that replication cannot?
A: Capacity and write throughput. Replication copies the whole dataset (helping availability and reads), but every copy is still the full size. Partitioning splits the dataset so it can exceed one machine's disk/RAM/write limits.
Q: Contrast range partitioning and hash partitioning on their two key trade-offs.
A: Range keeps keys ordered → efficient range scans (one shard) but clusters cause hot shards. Hash spreads keys evenly → balanced load and fast point lookups, but range queries become scatter-gather across all shards and range scans are lost.
Q: Why does a good hash function still leave you with hot shards?
A: Hashing balances distinct keys, not traffic. A single hot key (a celebrity account, a viral post) hashes to exactly one shard, so all its load lands there. Fixes are app-level: split the hot key with suffixes, or cache it hard in front.
Q: Why is hash(key) mod N a bad way to assign shards in practice?
A: When N changes (add/remove a node), the modulus changes for almost every key, so ~(N-1)/N of all data must move at once — a cluster-wide reshuffle. Use consistent hashing or a fixed large partition count so only ~1/N moves.
Q: Why is choosing the partition (shard) key the most important sharding decision?
A: It determines which queries stay single-shard (cheap) and which become scatter-gather (expensive), and whether load is balanced. It's very costly to change later because changing it re-partitions the entire dataset — so pick it for access patterns, not the data model.

If you only remember one thing: replication copies the whole dataset, partitioning cuts it into pieces so it can outgrow any single machine — and the instant you cut it, every request must answer "which piece holds this?" Judge any scheme on three tensions: is the load balanced (beware the hot key), how much data moves when you add a node (beware plain mod N), and can you find a row without asking every shard (choose the partition key for your access patterns). Master those, and tomorrow's consistent hashing will feel like the obvious cure it is.

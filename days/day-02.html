Horizontal vs Vertical Scaling

Every system eventually hits a wall. Traffic grows, queries slow down, servers buckle. How you respond to that wall — by making your single machine bigger, or by adding more machines — is one of the most consequential architectural decisions you'll make. Today you'll learn to make that choice deliberately and defend it in any interview.

Excalidraw — Vertical vs Horizontal Scaling

The Core Problem: Systems Under Load

When a system receives more traffic than it can handle, performance degrades. Response times increase, queues back up, and eventually the system fails entirely. This isn't a hypothetical — it's a predictable engineering challenge that every production system faces as it grows.

The question isn't if you'll need to scale — it's how. And the answer depends on what's actually bottlenecking your system.

The Four Classic Bottlenecks

Before you can scale, you need to identify what's limiting you:

Resource Symptom Measurement Tool
CPU High processing time, timeouts on compute-heavy operations top, htop, CloudWatch CPU metrics
Memory Swapping to disk, OOM kills, GC pressure free -h, heap profilers, memory dashboards
Disk I/O Slow database queries, write queuing, high iowait iostat, DB slow query logs, IOPS metrics
Network Packet loss, high latency, bandwidth saturation netstat, iperf, VPC flow logs
💡 Interview Insight

In interviews, when asked how you'd handle increased load, always start by identifying the bottleneck. "It depends on the bottleneck — CPU-bound vs memory-bound vs I/O-bound problems call for different solutions." This demonstrates senior-level thinking immediately.

Vertical Scaling (Scale Up)

Vertical scaling means making your existing machine more powerful. You replace the server with a bigger one — more CPU cores, more RAM, faster SSDs, higher-throughput network cards. The application doesn't change at all. You just throw more hardware at the problem.

The Hardware Upgrade Ladder

A typical vertical scaling progression for an application server:

t3.micro 2 vCPU, 1 GB RAM — startup MVP ~100 req/s
t3.large 2 vCPU, 8 GB RAM — early traction ~500 req/s
c6i.4xlarge 16 vCPU, 32 GB RAM — growing product ~3,000 req/s
c6i.32xlarge 128 vCPU, 256 GB RAM — serious scale ~25,000 req/s
u-24tb1.metal 448 vCPU, 24 TB RAM — AWS maximum Hard ceiling ⚠️
⚠️ The Hard Ceiling Problem

There's a physical limit to how large a single machine can be. Once you hit the largest available instance type (or the limit of your on-premises hardware budget), vertical scaling stops being an option. You're forced to either live with the constraint or shift to horizontal scaling — which is much harder to bolt on after the fact.

Advantages

  • Zero code changes required. Your application doesn't need to know it's on bigger hardware.
  • Simpler operations. No distributed systems complexity — one server, one database connection, no coordination.
  • Better for stateful workloads. In-memory caches, session state, and local locks just work.
  • Lower latency. No network hops between components — everything runs on the same machine.
  • ACID guarantees preserved. Single-machine databases maintain full transaction semantics without distributed coordination.

Disadvantages

  • Single point of failure. When the machine goes down, the entire service is down.
  • Hard ceiling. You cannot scale beyond the largest available machine size.
  • Expensive at the top end. The largest instances cost dramatically more per unit of compute than mid-range ones.
  • Downtime during upgrades. Resizing typically requires a restart, causing a service interruption.
  • Can't scale globally. You're stuck in one region, one datacenter, one machine.

Horizontal Scaling (Scale Out)

Horizontal scaling means adding more machines to your system. Instead of one powerful server, you have many smaller servers working in parallel. Traffic is distributed across all of them — typically via a load balancer sitting in front of your fleet.

This is the scaling model used by every major internet company. Netflix runs on tens of thousands of AWS instances. Google operates custom server farms with millions of commodity machines. Twitter's timeline service runs across hundreds of servers simultaneously.

How It Works

In a horizontally scaled system, a load balancer receives all incoming requests and distributes them across the server pool. Each server handles a subset of total traffic. When load increases, you add more servers. When it decreases, you remove them. (We'll cover load balancing algorithms in depth on Day 5.)

🔑 The Key Insight: Auto-Scaling

Horizontal scaling enables auto-scaling — automatically adding or removing servers based on real-time load. AWS Auto Scaling, Kubernetes Horizontal Pod Autoscaler (HPA), and GCP Managed Instance Groups all do this. When CPU hits 70%, spin up 2 more nodes. When it drops to 20%, scale back down. This is essentially impossible with vertical scaling.

Advantages

  • No hard ceiling. In theory, you can keep adding machines indefinitely.
  • High availability by default. When one server fails, others continue serving traffic.
  • Auto-scaling. Cloud platforms can add capacity in response to real-time demand.
  • Geographic distribution. You can deploy across multiple regions for global low-latency access.
  • Cost-efficient at scale. Many cheap commodity servers can be more cost-effective than one massive machine.
  • Rolling deployments. Update servers one at a time without downtime.

Disadvantages

  • Statelessness is required. If one server stores session data locally, other servers can't access it. (More on this below.)
  • Distributed systems complexity. Consistency, consensus, and coordination become hard problems.
  • Harder debugging. Logs are spread across many machines; tracing a request requires distributed tracing.
  • Network overhead. Inter-service communication adds latency that doesn't exist in a single machine.
  • More infrastructure to manage. Load balancers, service discovery, health checks, deployment pipelines — the ops burden is higher.

The Stateless Requirement

This is the most important concept for understanding why horizontal scaling is hard in practice. Horizontal scaling requires stateless application servers. If you don't understand statelessness, you can't design scalable systems.

What Is State?

"State" is any data that persists between requests. In a simple web application, this includes:

  • User sessions — who is logged in, what's in their cart
  • In-memory caches — data loaded from the database and held in RAM
  • Request context — ongoing file uploads, streaming connections
  • Local files — uploaded images stored on the server's local disk

The "Sticky Session" Anti-Pattern

Suppose a user logs in and their session is stored in Server A's memory. On their next request, the load balancer routes them to Server B. Server B has no knowledge of the session — the user appears logged out. This is the sticky session problem.

The naive "fix" is sticky sessions — the load balancer always routes the same user to the same server. This breaks horizontal scaling: one server now has more load than others, and if that server dies, all users on it lose their sessions.

🚫 The Sticky Session Anti-Pattern

Sticky sessions tie user traffic to specific servers. This defeats load balancing, prevents proper failover, and creates hot spots. It's a Band-Aid that makes the real problem worse. Never design a new system that relies on sticky sessions.

The Right Solution: Externalize State

The correct approach is to move all state out of the application servers and into a shared, externalized store. Application servers become stateless workers — any server can handle any request because all persistent data lives outside them.

State Type Bad (Local) Good (Externalized)
User sessions In-memory HashMap on the server Redis / Memcached cluster
Uploaded files Server local disk at /uploads/ S3, GCS, or Azure Blob Storage
Application data SQLite file on the server RDS, Aurora, or Cloud Spanner
Feature flags Config file on each server LaunchDarkly, AWS AppConfig
Computed results Local in-process cache Redis with TTL

Once state is externalized, any server in your fleet can serve any request. The load balancer can distribute traffic freely. Servers can be added or removed without affecting users. This is what "cloud-native" actually means in practice.

Database Scaling: The Hard Part

Application servers are easy to scale horizontally because they can be made stateless. Databases are fundamentally stateful — they store data. This is why database scaling is the hard part of every system design, and why it gets dedicated attention on Days 11–14.

For now, understand the two primary approaches:

Scale Up

Vertical DB Scaling

Upgrade to a more powerful database server. More RAM means more data fits in buffer pool. Faster CPUs handle more concurrent queries. Better SSDs reduce I/O latency.

✓ Works for
  • Early-stage systems
  • Write-heavy workloads
  • Strong consistency needs
✗ Limits
  • Hard hardware ceiling
  • Expensive at top end
  • Single point of failure
Scale Out

Horizontal DB Scaling

Split data across multiple database nodes. Read replicas for read-heavy workloads. Sharding for massive write-heavy workloads. Dramatically increases complexity.

✓ Works for
  • Read-heavy workloads
  • Massive data volumes
  • High write throughput
✗ Limits
  • Cross-shard queries are hard
  • Replication lag
  • Much harder to manage

Read Replicas: The First Database Scale-Out Step

The most common first step in database horizontal scaling is read replicas. You keep one primary (write) node and create multiple replicas that handle reads. Since most web applications are read-heavy (10:1 to 100:1 read/write ratio is common), this dramatically increases throughput.

The tradeoff: replication lag. Replicas are slightly behind the primary. A write on the primary might not be visible on replicas for 10–100ms. For most reads this is fine. For reads that must see the latest write (e.g., "did my payment go through?"), you route to the primary. We'll cover replication in depth on Day 12.

Side-by-Side Comparison

Dimension Vertical (Scale Up) Horizontal (Scale Out)
Scale limit Hard ceiling (largest machine) Theoretically unlimited
Fault tolerance Single point of failure High availability by default
Code changes None required Must be stateless
Ops complexity Low High (load balancers, discovery, etc.)
Downtime Required for resize Zero downtime (rolling deploy)
Cost model Single large instance (expensive) Many small instances (efficient at scale)
Auto-scaling Limited / disruptive Native, sub-minute elasticity
Multi-region Not possible Native support
Best for Databases, legacy apps, early stage Stateless services, APIs, web servers

When to Use Each: The Decision Framework

The choice isn't always one or the other. Most production systems use vertical scaling for some components and horizontal scaling for others. Use this mental model:

Scenario → Recommended Approach

You're a startup with <10k daily active users Scale Up First
Your web/API servers are CPU-bound under load Scale Out
Your database is the bottleneck Scale Up + Read Replicas
You need zero-downtime deployments Scale Out
Your app maintains per-request in-memory state Scale Up (or refactor first)
You're serving users across multiple continents Scale Out (multi-region)
Traffic spikes 10× during events (sports, product launches) Scale Out (auto-scaling)
You're running a monolithic legacy app you can't refactor Scale Up + Strangler Fig Pattern
✅ The Pragmatic Engineer's Rule

Scale vertically first. It's simpler, faster to implement, and often all you need. Only move to horizontal scaling when you hit the ceiling or need high availability. The biggest mistake engineers make is over-engineering for scale that never comes. Instagram had 13 engineers when it was acquired by Facebook for $1B — they ran on relatively few vertically-scaled machines until they actually needed to scale out.

Real-World Examples

📺 Netflix

Netflix runs entirely on AWS and is a canonical example of horizontal scaling. Their microservices architecture runs thousands of service instances simultaneously. Their API gateway alone runs on hundreds of nodes. They pioneered chaos engineering (Chaos Monkey) specifically because they needed to verify their horizontal architecture was truly resilient to node failures. Database scaling is handled via a combination of Cassandra (horizontally scaled NoSQL) and MySQL with read replicas for relational data.

🐦 Twitter / X

Twitter's early architecture was a monolithic Ruby on Rails application that scaled primarily vertically. By 2008–2010, this became unsustainable — the "Fail Whale" era. They rebuilt their timeline service using a horizontally scaled Scala/Finagle stack. The timeline fan-out service eventually ran on hundreds of nodes writing to Redis clusters. This migration took years and is one of the most famous scaling rewrites in industry history.

🛒 Amazon

Amazon's move from a monolith to services (which became AWS) was driven by horizontal scaling needs. Their "2 pizza teams" rule and service-oriented architecture allowed independent horizontal scaling per service. Today, the Amazon.com product page alone calls 100+ independent microservices, each horizontally scaled independently. Their databases scaled through a combination of read replicas, DynamoDB (horizontally scaled NoSQL), and Aurora for relational workloads.

📷 Instagram

Instagram's early architecture is instructive. They started with Django (Python) on a single EC2 instance, then vertically scaled as they grew. They added PostgreSQL read replicas early. Even at 1 million users, they were running a relatively small number of machines. Their clever use of vertical scaling bought them time to build the stateless architecture needed for proper horizontal scaling later. By the time they needed it, they could do it right.

📌 Pattern: The Vertical-First Startup

Notice the pattern: most successful startups start vertical, prove product-market fit, then invest in horizontal scaling infrastructure when the traffic actually demands it. Horizontal scaling is an investment in operational complexity. Make it when you must, not before.

Using This in Interviews

Scaling questions come up constantly in system design interviews. Here's how to apply today's concepts.

When the Interviewer Says "Handle 1M Users"

Don't immediately jump to Kubernetes and auto-scaling. Walk through the logic:

  1. Estimate the actual request rate (Day 1 framework: 1M daily active users × 10 requests/day / 86400 seconds ≈ 116 req/s)
  2. A single well-tuned server handles 1,000–10,000 req/s. You don't need horizontal scaling yet.
  3. But you need high availability — so minimum 2 servers behind a load balancer.
  4. The database is usually the first bottleneck — add a read replica before you add app servers.

When the Interviewer Says "Handle 100M Users"

Now horizontal scaling is genuinely required:

  1. Application tier: stateless servers behind a load balancer, auto-scaled per CPU utilization
  2. Session state: externalized to Redis cluster
  3. Static assets: CDN (Day 4)
  4. Database: read replicas + consider sharding (Day 13) or switching to NoSQL (Day 8)
  5. Consider microservices to scale individual bottleneck services independently (Day 77)
⚠️ Common Interview Mistake

Don't recommend horizontal scaling as your first answer to every scaling question. Interviewers look for calibrated judgment. If someone asks you to design a system for 10,000 users and you immediately propose Kubernetes with 50 microservices, that's a red flag — it shows you can't match solution complexity to problem complexity.

✅ Key Takeaways for Day 2

  • Vertical scaling = bigger machine. Simple, no code changes, but has a hard ceiling and is a single point of failure.
  • Horizontal scaling = more machines. Theoretically unlimited, high availability, but requires stateless application design.
  • Statelessness is the key to horizontal scaling — externalize all state to Redis, S3, or a database, never store it on the application server.
  • Databases are harder to scale horizontally than application servers. Start with read replicas, escalate to sharding only when needed.
  • Scale vertically first in practice — it's simpler and often sufficient until you actually have the traffic to justify distributed systems complexity.
  • In interviews, always justify your scaling choice with estimated request rates, not by defaulting to "use Kubernetes."

🧠 Practice Questions

1
You're designing a web application that starts with 1,000 users and needs to grow to 10 million. Describe your scaling strategy at each order of magnitude: 1K, 10K, 100K, 1M, and 10M users. What changes at each stage?
Hint: Start with a single server. Add a database. Add a replica. Add servers. Add caching. Consider sharding. Each step should be justified by actual bottlenecks.
2
A shopping cart feature stores items in a HashMap on each application server. Traffic grows and you want to add more servers. What problem will you encounter? How do you fix it without changing the user experience?
Hint: Think about what happens when the load balancer sends the user to a different server for their next request.
3
Your API server's CPU is at 90% but your database CPU is at 20%. You need to reduce latency. Would you scale the API servers vertically, horizontally, or should you investigate the database? What questions would you ask?
Hint: Is the API CPU-bound due to computation, or is it waiting for I/O from the database? Profiling reveals the real bottleneck.
4
Twitter's engineering team reported that at peak load, a single tweet from a major celebrity (like Obama) would trigger 10+ million timeline updates within seconds. Why does this make horizontal scaling essential, and what would break if they tried to handle this with a single large server?
Hint: Think about the write amplification problem — one write creates millions of derivative writes.
5
An interviewer asks you to "design a system that scales to 1 billion users." Before designing anything, what 5 questions would you ask to understand whether vertical or horizontal scaling is the right starting point?
Hint: Apply Day 1's framework. What are the functional requirements? What's the read/write ratio? What does "scale" mean here — user count, request rate, or data volume?
